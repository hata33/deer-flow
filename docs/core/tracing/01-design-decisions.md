# 01 - 设计决策与考量

> 本文档回答"为什么这么做"——每个设计决策的动机、解决的问题、权衡取舍。

---

## 一、核心设计决策清单

| # | 决策 | 一句话动机 |
|---|------|-----------|
| 1 | **可观测性优于调试** | 分布式 Agent 系统中 print/logging 不够用 |
| 2 | **结构化事件而非文本日志** | 机器可解析，支持过滤和聚合 |
| 3 | **按需启用而非始终记录** | 避免性能开销和存储膨胀 |
| 4 | **与 LangSmith/OpenTelemetry 兼容** | 利用现有生态，不造新轮子 |

---

## 二、逐决策分析

### 决策 1：可观测性优于调试

**问题**: Agent 系统的执行路径不确定——LLM 的输出、工具的选择、中间件的行为都不可预测。传统断点调试效率极低。

**选择**: 通过结构化事件追踪执行过程。记录每次 LLM 调用的输入/输出、每次工具调用的参数/结果、中间件的决策过程。

### 决策 2：结构化事件

**问题**: 文本日志难以过滤、聚合和可视化。

**选择**: 事件包含结构化字段（run_id、thread_id、timestamp、event_type、duration 等），支持 JSON 输出和下游分析。

### 决策 3：按需启用

**问题**: 全量追踪会显著增加延迟和存储开销。

**选择**: 追踪默认关闭，通过配置按需启用。只在排查问题时打开，不影响正常运行性能。

### 决策 4：生态兼容

**问题**: 自建追踪系统成本高、维护难。

**选择**: 与 LangSmith 和 OpenTelemetry 协议兼容，用户可以选择自己的观测后端。
