# 01 - 设计决策与考量

> 本文档回答"为什么这么做"——每个设计决策的动机、解决的问题、权衡取舍。

---

## 一、核心设计决策清单

| # | 决策 | 一句话动机 |
|---|------|-----------|
| 1 | **Readability 算法提取正文** | 去除导航栏/广告噪音，保留核心内容 |
| 2 | **HTML → Markdown 转换** | LLM 对 Markdown 的理解远优于 HTML |
| 3 | **pymupdf4llm 优先 + markitdown 回退** | PDF 转换质量第一，兼容性兜底 |
| 4 | **4096 字符硬截断** | 简单有效地控制上下文长度 |
| 5 | **文件名清洗 normalize_filename** | 防止路径穿越攻击 |

---

## 二、逐决策分析

### 决策 1：Readability 算法

**问题**: 网页 HTML 包含大量噪音（导航栏、侧边栏、广告、页脚）。直接喂给 LLM 浪费 token 且干扰理解。

**选择**: Mozilla 的 Readability 算法（与 Firefox 阅读模式相同）提取正文。配合 python-readability 库实现。

### 决策 2：HTML → Markdown

**问题**: LLM 对 HTML 标签的理解不如 Markdown 格式。Markdown 更紧凑，token 效率更高。

**选择**: 使用 markdownify 或类似库将 HTML 转换为 Markdown。保留标题层级、链接、列表等语义结构。

### 决策 3：PDF 双引擎

**问题**: PDF 格式复杂（扫描件、矢量图、表格、多栏布局），单一引擎难以覆盖所有情况。

**选择**: pymupdf4llm（基于 PyMuPDF）优先——对文本 PDF 质量最高。如果输出稀疏（<50 字符/页），回退到 markitdown（基于 markitdown 库，兼容性更好）。

### 决策 4：硬截断

**问题**: 不同 LLM 的上下文窗口差异巨大，工具无法知道调用方的窗口大小。

**选择**: 固定 4096 字符截断。简单、可预测、适用于大多数场景。调用方（中间件/LLM）负责进一步的上下文管理。

### 决策 5：文件名清洗

**问题**: 用户上传的文件名可能包含 `../`、`\0`、绝对路径等危险字符。

**选择**: `normalize_filename()` 移除路径分隔符、控制字符，只保留安全字符。防御路径穿越攻击的第一道防线。
