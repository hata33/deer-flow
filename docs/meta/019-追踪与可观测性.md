# 追踪与可观测性

**问题**: Agent 系统的执行路径不可预测（LLM 自主决策调用什么工具），传统监控看不懂。需要专门的追踪系统记录"Agent 做了什么、为什么这么做、花了多少 token"。

---

## 问题 1：支持哪些追踪平台？

| 平台 | 特点 | 配置 |
|------|------|------|
| LangSmith | LangChain 生态原生 | 最简单 |
| Langfuse | 开源、可自部署 | 支持多实例 |

```yaml
tracing:
  enabled: true
  providers:
    langsmith:
      api_key: "${LANGSMITH_API_KEY}"
      project: "deer-flow-prod"
    langfuse:
      public_key: "${LANGFUSE_PUBLIC_KEY}"
      secret_key: "${LANGFUSE_SECRET_KEY}"
      host: "https://cloud.langfuse.com"  # 或自部署地址
```

可以同时启用多个平台——事件会发送到所有已配置的追踪器。

---

## 问题 2：追踪什么内容？

| 追踪项 | 内容 | 用途 |
|--------|------|------|
| LLM 调用 | 模型、prompt、输出、token 数 | 成本分析、质量审计 |
| 工具调用 | 工具名、参数、结果、耗时 | 调试、安全审计 |
| Agent 循环 | 每轮的输入/输出 | 理解 Agent 决策过程 |
| 子 Agent | 子 Agent 的完整执行轨迹 | 端到端追踪 |
| 错误 | 异常类型、消息、堆栈 | 故障排查 |

```
Run (root trace)
    ├── LLM Call #1 (span)
    │   ├── input: 500 tokens
    │   └── output: 200 tokens
    ├── Tool Call: bash (span)
    │   ├── args: {"command": "ls -la"}
    │   └── result: "file1.txt\nfile2.txt"
    ├── LLM Call #2 (span)
    └── Tool Call: write_file (span)
```

---

## 问题 3：对性能有影响吗？

追踪是**延迟初始化**的：

```python
# 不追踪时零开销
if not tracing_enabled:
    return  # 直接返回，不导入 SDK
```

只有在 `tracing.enabled: true` 时才导入 SDK 和注册回调。禁用时没有任何性能损失。

---

## 问题 4：追踪 SDK 怎么加载？

通过反射动态导入，不硬编码依赖：

```python
# tracing/factory.py
def setup_tracing(config):
    for provider_name in config.providers:
        if provider_name == "langsmith":
            import langsmith  # 延迟导入
            langsmith.configure(...)
        elif provider_name == "langfuse":
            from langfuse.callback import CallbackHandler
            handler = CallbackHandler(...)
```

没装 `langsmith` 包？没关系，不启用就不会导入。

---

## 问题 5：追踪数据怎么关联到 Run？

通过 `RunJournal` 桥接：

```
LLM 调用
    ├── RunJournal 记录 → 数据库（RunEventRow）
    └── CallbackHandler → 追踪平台（LangSmith/Langfuse）
```

两个系统独立运行但共享相同的 Run ID，可以交叉查询：
- 在追踪平台看到 token 异常 → 用 Run ID 在数据库查事件详情
- 在数据库看到错误 → 在追踪平台查完整调用链

---

## 问题 6：错误会被追踪吗？

会。追踪系统记录所有异常，包括：

- LLM API 错误（限流、超时）
- 工具执行异常
- 中间件错误

```python
# 追踪自动捕获异常
try:
    result = await llm.ainvoke(prompt)
except Exception as e:
    # 追踪平台自动记录 span 状态 = error
    raise
```

错误不会被吞掉——追踪只记录，不干预执行流程。

---

## 问题 7：生产环境怎么用追踪？

典型场景：

| 场景 | 操作 | 工具 |
|------|------|------|
| 成本监控 | 统计 token 消耗趋势 | LangSmith Dashboard |
| 质量审计 | 检查 Agent 的工具调用是否合理 | Langfuse Trace |
| 故障排查 | 查看失败 Run 的完整调用链 | 两个平台都行 |
| 性能优化 | 找到最耗时的工具/LLM 调用 | Latency 分析 |

建议：生产环境至少启用一个追踪平台。

---

## 问题 8：敏感信息怎么处理？

LLM 输入和输出可能包含敏感数据（用户对话、文件内容）。追踪平台会完整记录这些内容。

防护措施：
- 使用自部署的 Langfuse 实例，数据不出内网
- LangSmith 支持配置数据保留策略
- 可以在追踪回调中过滤敏感字段

```yaml
tracing:
  providers:
    langfuse:
      host: "https://langfuse.internal.company.com"  # 内网部署
      mask_inputs: true   # 遮蔽敏感输入
```

## 数据流概览

```
Agent 执行
    │
    ├── LangChain Callbacks
    │   ├── RunJournal → 数据库（事件流）
    │   └── Tracing Callback → 追踪平台
    │       ├── LangSmith (云端)
    │       └── Langfuse (自部署)
    │
    ▼ 追踪数据结构
Trace
    ├── Span: LLM Call #1
    │   ├── input tokens: 500
    │   └── output tokens: 200
    ├── Span: Tool Call bash
    │   ├── duration: 1.2s
    │   └── status: ok
    └── Span: LLM Call #2
        └── status: error (RateLimitError)
```
