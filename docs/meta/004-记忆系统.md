# 记忆系统

**问题**: Agent 每次对话都是"失忆"的，无法记住用户偏好、工作上下文和历史决策，导致重复提问和体验割裂。

---

## 问题 1：记忆存什么？

四类结构化数据：

| 类别 | 内容 | 更新频率 |
|------|------|---------|
| **user** | 工作上下文、个人信息、当前关注点 | 每次对话更新 |
| **history** | 近期摘要、早期背景、长期记忆 | 对话结束时更新 |
| **facts** | 具体事实（偏好、知识、习惯） | 实时提取 |
| **version** | 记忆格式版本号 | 极少变化 |

```json
{
  "version": "1.0",
  "user": {
    "workContext": {"summary": "后端工程师，负责支付系统"},
    "personalContext": {"summary": "偏好 TypeScript，用 Neovim"},
    "topOfMind": {"summary": "正在迁移数据库到 PostgreSQL"}
  },
  "facts": [
    {"content": "用户偏好暗色主题", "confidence": 0.95, "category": "preference"}
  ]
}
```

---

## 问题 2：记忆怎么提取？

LLM 在对话过程中被动提取，不需要用户显式声明。`memory_updater` 分析每轮对话，识别出用户偏好、关键事实、工作上下文等信息，自动写入记忆。

```
用户: "帮我用 Python 写个脚本，别用 requests，我喜欢 httpx"
                    │
                    ▼ LLM 分析
提取事实: "用户偏好 httpx 库" (confidence: 0.9, category: preference)
                    │
                    ▼ 写入记忆
facts 列表新增一条
```

---

## 问题 3：记忆什么时候更新？实时还是延迟？

**延迟更新**。记忆更新有 30 秒防抖：

```
对话中提取到新事实
    │
    ▼ add_nowait() 放入队列
等待 30 秒（防抖）
    │
    ▼ 无新提取 → 批量写入记忆
    │
    ▼ 30 秒内又有新提取 → 重置计时器
```

为什么防抖：频繁写入会增加 LLM 调用次数和存储压力，30 秒内的事实合并处理更高效。

**例外**：上下文压缩时，即将删除的消息会触发 `add_nowait` 并跳过防抖（因为消息马上要没了，等不及）。

---

## 问题 4：记忆怎么注入到对话？

`DynamicContextMiddleware` 在每次 LLM 调用前，将记忆格式化后注入系统提示。

注入有 token 预算限制（默认 2000 token），超出时按置信度优先级截断：

```
记忆中有 80 条 facts
    │
    ▼ 按置信度降序排列
从高到低逐条添加
    │
    ▼ 累计 token 接近 2000
停止添加，低置信度的 facts 被省略
    │
    ▼ 结果
Agent 看到的是"精华子集"，不是全部记忆
```

---

## 问题 5：记忆会不会无限膨胀？

三重防线控制增长：

| 防线 | 限制 | 机制 |
|------|------|------|
| facts 数量上限 | 最多 100 条 | 超出按置信度降序截断 |
| 置信度门槛 | ≥ 0.7 | 低于门槛直接丢弃 |
| 注入 token 预算 | 最多 2000 token | 注入时截断 |

```yaml
memory:
  max_facts: 100
  fact_confidence_threshold: 0.7
  max_injection_tokens: 2000
```

---

## 问题 6：记忆存在哪里？

默认使用文件存储（`FileMemoryStorage`），每个用户一个 JSON 文件：

```yaml
memory:
  storage_class: "deerflow.agents.memory.storage.FileMemoryStorage"
  storage_path: ""  # 默认 .deer-flow/memory/
```

文件结构：
```
.deer-flow/memory/
    └── {user_id}/
        └── memory.json
```

可替换为数据库存储——只需实现 `MemoryStorage` 接口。

---

## 问题 7：记忆会被错误信息污染吗？

两层过滤：

1. **置信度门槛**：LLM 提取事实时打置信度分，低于 0.7 直接丢弃
2. **summary 覆盖而非追加**：user/history 字段是覆盖式更新，不是追加式，避免过时信息累积

如果 LLM 判断某个事实不再成立（置信度低于门槛），它自然不会被保留。

---

## 问题 8：多线程/多会话并发访问记忆怎么处理？

文件存储使用文件锁保证一致性。记忆更新是原子操作——读取→合并→写回，整个过程在锁的保护下完成。

---

## 数据流概览

```
用户对话
    │
    ▼ LLM 调用前
DynamicContextMiddleware 注入记忆（2000 token 预算）
    │
    ▼ Agent 回复
memory_updater 分析对话，提取事实
    │
    ▼ 放入队列，30 秒防抖
    │
    ▼ 防抖到期
批量合并写入 memory.json
    │
    ▼ 下次对话
从 memory.json 读取，注入 Agent
```
