# LLM错误处理与重试

**问题**: LLM API 调用在生产环境中有 1-5% 的失败率（限流、超时、服务端错误），如果每次失败都直接崩溃，Agent 根本无法完成复杂任务。

---

## 问题 1：LLM 调用会出什么错？

| 错误类型 | 原因 | 频率 |
|---------|------|------|
| `RateLimitError` | 请求超过 API 配额 | 高频 |
| `APITimeoutError` | 网络延迟或服务端慢 | 中频 |
| `APIConnectionError` | 网络中断 | 低频 |
| `InternalServerError` | LLM 提供商故障 | 低频 |
| `BadRequestError` | 请求参数错误（如 token 超限） | 中频 |

生产经验：**不是会不会出错，而是什么时候出错**。必须为每种错误准备对策。

---

## 问题 2：重试策略是什么？

`llm_error_handling_middleware` 实现了指数退避重试：

```
第 1 次失败 → 等待 1 秒 → 重试
第 2 次失败 → 等待 2 秒 → 重试
第 3 次失败 → 等待 4 秒 → 重试
    ...
达到最大重试次数 → 触发熔断
```

不是所有错误都重试：

| 错误类型 | 是否重试 | 原因 |
|---------|---------|------|
| `RateLimitError` | 重试 | 限流是临时的 |
| `APITimeoutError` | 重试 | 可能是暂时延迟 |
| `APIConnectionError` | 重试 | 网络可能恢复 |
| `BadRequestError` | 不重试 | 请求本身有问题，重试没用 |
| `InternalServerError` | 重试 | 提供商可能恢复 |

---

## 问题 3：熔断器（Circuit Breaker）怎么工作？

重试次数用尽后启动熔断器，防止继续打已经挂掉的服务：

```
状态机：
Closed（正常）→ 连续失败达到阈值 → Open（熔断）
                                          │
                                     等待冷却时间
                                          │
Open → Half-Open（半开）→ 尝试一次请求
                              ├── 成功 → Closed
                              └── 失败 → Open
```

| 参数 | 含义 | 默认值 |
|------|------|-------|
| 失败阈值 | 连续失败几次触发熔断 | 5 |
| 冷却时间 | 熔断后等多久再试 | 60 秒 |
| 最大重试 | 单次请求最大重试次数 | 3 |

---

## 问题 4：重试期间 Agent 的状态会丢吗？

不会。`wrap_model_call` 钩子包裹整个 LLM 调用过程，重试是内部行为——Agent 无感知。重试期间对话状态完整保存在 LangGraph 检查点中。

```
Agent 发起 LLM 调用
    │
    ▼ wrap_model_call 钩子
    │
    ▼ 调用 LLM API
    ├── 成功 → 返回结果
    └── 失败 → 等待 → 重试（Agent 不知道）
              ├── 成功 → 返回结果
              └── 耗尽重试 → 触发熔断 → 返回错误
```

---

## 问题 5：工具执行错误怎么处理？

工具错误和 LLM 调用错误是两套机制。`tool_error_handling_middleware` 在 `wrap_tool_call` 钩子中捕获工具异常：

```python
# 策略：转换为 ToolMessage，不终止 Run
try:
    result = await tool.ainvoke(input)
except Exception as e:
    error_msg = f"Error: Tool '{tool_name}' failed with {type(e).__name__}: {str(e)}"
    return ToolMessage(content=error_msg[:500], status="error")
```

关键设计：**错误变成消息，不是异常**。Agent 看到错误后可以：
1. 换个参数重试
2. 换个工具
3. 向用户报告问题

---

## 问题 6：悬空工具调用（Dangling Tool Call）是什么？

用户在工具执行期间中断（发送新消息），导致 LLM 的 `tool_calls` 没有对应的 `ToolMessage`。LangGraph 要求每个 `tool_call` 都有对应的 `ToolMessage`，否则报错。

`dangling_tool_call_middleware` 自动修复：

```
LLM: tool_calls=[{id: "call_1", name: "bash", args: {...}}]
用户中断 → bash 结果丢失
    │
    ▼ dangling_tool_call_middleware 检测
    │
    ▼ 补一个 ToolMessage
ToolMessage(id="call_1", content="Tool execution was interrupted by user")
```

---

## 问题 7：before_model 和 wrap_model_call 的区别？

两种钩子解决不同层面的问题：

| 维度 | before_model | wrap_model_call |
|------|-------------|----------------|
| 时机 | LLM 调用前 | 包裹整个 LLM 调用 |
| 能否修改请求 | 能（如压缩上下文） | 能 |
| 能否处理错误 | **不能** | 能 |
| 能否阻止调用 | 不能 | 能（不调 handler） |
| 典型用途 | 注入上下文、压缩 | 错误重试、熔断 |

---

## 问题 8：错误信息会泄露敏感信息吗？

不会。错误消息做了截断处理（最多 500 字符），且只包含异常类型和消息文本，不包含堆栈跟踪、请求参数等敏感信息。

---

## 数据流概览

```
Agent 请求 LLM 调用
    │
    ▼ before_model 钩子
压缩上下文 / 注入 todo / 注入图片描述
    │
    ▼ wrap_model_call 钩子
    │
    ▼ 调用 LLM API
    ├── RateLimitError → 指数退避重试
    ├── TimeoutError → 重试
    ├── BadRequestError → 直接返回错误
    └── 成功 → 返回结果
    │
    ▼ 工具调用
    ├── 正常执行 → ToolMessage
    └── 异常 → ToolMessage(status="error")
    │
    ▼ 悬空检测
缺失 ToolMessage → 自动补充
```
