# 护栏系统

**问题**: Agent 可以调用各种工具（bash、文件操作、网络请求等），需要策略层控制"哪些工具能用、什么条件下能用"，防止误操作或恶意使用。

---

## 问题 1：护栏和沙箱有什么区别？

两层安全，各管各的：

| 维度 | 护栏（Guardrails） | 沙箱（Sandbox） |
|------|-------------------|----------------|
| 作用层 | 工具调用前拦截 | 文件系统隔离 |
| 关注点 | "能不能用这个工具" | "能访问哪些路径" |
| 举例 | 禁止 `rm` 命令 | `/mnt/user-data/` 之外的路径不可写 |
| 时机 | 工具调用前（`before_tool`） | 工具执行时 |

两者互补：护栏决定"能不能调"，沙箱决定"调了之后在什么范围内操作"。

---

## 问题 2：护栏的决策流程是什么？

```
Agent 调用 tool("bash", {"command": "rm -rf /"})
    │
    ▼ guardrails_middleware 拦截（before_tool 钩子）
构建 GuardrailRequest
    │
    ▼ 调用 GuardrailProvider
    ├── AllowlistProvider → 检查工具是否在白名单
    ├── OAP Provider → 调用外部策略 API
    └── 自定义 Provider → 自定义逻辑
    │
    ▼ 返回 GuardrailDecision
    ├── allow=True → 放行，工具执行
    └── allow=False → 拒绝，返回拒绝原因给 Agent
```

---

## 问题 3：GuardrailRequest 和 GuardrailDecision 长什么样？

```python
@dataclass
class GuardrailRequest:
    tool_name: str          # 工具名，如 "bash"
    tool_input: dict        # 工具参数，如 {"command": "rm -rf /"}
    agent_id: str           # Agent 标识
    timestamp: datetime     # 调用时间

@dataclass
class GuardrailDecision:
    allow: bool             # 是否允许
    reasons: list[GuardrailReason]  # 决策原因
    policy_id: str          # 匹配的策略 ID
```

## 问题 4：内置的 AllowlistProvider 怎么配置？

```yaml
guardrails:
  enabled: true
  fail_closed: true  # 策略评估失败时默认拒绝
  provider: deerflow.guardrails.builtin:AllowlistProvider
  config:
    allowed_tools: ["bash", "read_file", "write_file", "web_search"]
    denied_tools: ["rm", "dd", "curl", "wget"]
```

`denied_tools` 优先于 `allowed_tools`——即使 `bash` 在白名单中，如果 bash 命令包含 `rm`，仍可能被拒绝。

---

## 问题 5：fail_closed 是什么意思？

`fail_closed: true` 表示如果策略评估出错（网络超时、Provider 崩溃等），**默认拒绝**。

| fail_closed | 评估失败时 | 适用场景 |
|-------------|-----------|---------|
| `true` | 拒绝调用 | 生产环境（安全优先） |
| `false` | 放行调用 | 开发环境（可用性优先） |

---

## 问题 6：能接入外部策略引擎吗？

能。通过自定义 Provider 实现对接外部 OPA（Open Policy Agent）或自定义 API：

```yaml
guardrails:
  enabled: true
  provider: "my_company.custom:MyPolicyProvider"
  config:
    api_endpoint: "https://policy-api.example.com"
    timeout_seconds: 5
```

只需实现 `GuardrailProvider` 接口的 `evaluate(request) -> decision` 方法。

---

## 问题 7：护栏和技能的工具策略怎么协作？

两层过滤的交集：

```
所有已注册工具
    │
    ▼ 护栏过滤（全局策略）
    │
    ▼ 技能工具策略（技能级过滤）
    │
    ▼ 最终可用工具列表
```

一个工具必须**同时**通过两层过滤才能被 Agent 使用。技能允许但护栏拒绝的 → 不能用。护栏允许但技能不允许的 → 也不能用。

---

## 问题 8：被拒绝的工具调用 Agent 会怎样？

Agent 收到包含拒绝原因的错误消息，可以：
1. 换一个工具完成相同任务
2. 调用 `ask_clarification` 向用户说明情况
3. 放弃当前操作，报告失败

不会被拒绝"卡死"——Agent 有替代方案。

---

## 数据流概览

```
Agent 选择工具 bash(rm -rf /)
    │
    ▼ guardrails_middleware（before_tool）
    │
    ▼ GuardrailProvider.evaluate()
    │
    ├── 允许 → 工具执行 → 结果返回 Agent
    │
    └── 拒绝 → 原因返回 Agent
                  │
                  ▼ Agent 调整策略
              换工具 / 澄清 / 报告失败
```
