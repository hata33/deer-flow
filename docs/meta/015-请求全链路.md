# 请求全链路

**问题**: 一个用户请求从 HTTP 到达 Gateway，经过 RunManager、LangGraph、20 个中间件，最终通过 SSE 流式返回——中间任何一环出问题都需要可追踪。

---

## 问题 1：请求经过哪些模块？

```
用户发送消息
    │
    ▼ ① Gateway HTTP 入口
thread_runs.py 接收 POST /api/threads/{id}/runs
    │
    ▼ ② RunManager 创建 Run
    │
    ▼ ③ Agent 构建（20 个中间件装配）
    │
    ▼ ④ LangGraph 执行 Agent 循环
    │   ├── before_model 钩子链
    │   ├── LLM 调用
    │   ├── after_model 钩子链
    │   ├── before_tool 钩子链
    │   ├── 工具执行
    │   └── after_tool 钩子链
    │
    ▼ ⑤ SSE 流式推送
    │
    ▼ ⑥ 前端渲染
```

---

## 问题 2：20 个中间件的执行顺序？

中间件分两类：**基础中间件**（必选）和**条件中间件**（按配置启用）。

执行顺序（before_model 钩子为例）：

```
基础（8 个）:
  DynamicContext → Summarization → Todo → ViewImage → ...

条件（最多 12 个）:
  Memory → Sandbox → Guardrails → Upload → LoopDetection → ...
```

关键规则：
- Summarization 在 Todo **之前**（先压缩再注入，避免 todo 被压缩掉）
- Guardrails 在工具执行**之前**（拦截不合规调用）
- LoopDetection 在工具执行**之后**（统计调用模式）

---

## 问题 3：SSE 事件格式是什么？

四种事件类型：

| 类型 | 用途 | 示例 |
|------|------|------|
| `values` | LangGraph 状态快照 | 完整对话历史 |
| `messages-tuple` | 增量消息 | 新增的 AIMessage |
| `custom` | 自定义事件 | 中间件通知、进度更新 |
| `end` | 流结束 | Run 完成/失败 |

```
event: messages-tuple
data: {"messages": [{"type": "ai", "content": "我来帮你..."}]}

event: custom
data: {"type": "tool_call", "tool": "bash", "args": {"command": "ls"}}

event: end
data: {"status": "success", "run_id": "run_abc123"}
```

---

## 问题 4：StreamBridge 是什么？

`StreamBridge` 是 Run 和前端之间的消息总线：

```
LangGraph Agent ←→ StreamBridge ←→ SSE 连接 ←→ 前端
```

- **生产者**：Agent 执行过程中产生事件
- **消费者**：SSE 连接持续读取事件
- **事件 ID**：`{timestamp}-{sequence}` 格式，保证有序

```python
class StreamBridge:
    async def publish(self, event):
        """Agent 侧发布事件"""

    async def consume(self):
        """SSE 侧消费事件"""
```

两种实现：
- `MemoryStreamBridge`：内存队列，保留最近 256 个事件
- Gateway 模式：直接对接 HTTP SSE

---

## 问题 5：前端怎么处理流式响应？

`frontend/src/core/threads/hooks.ts` 中的 `onUpdateEvent` 回调处理 SSE 事件：

```
SSE 事件到达
    │
    ▼ 根据 event type 分发
    ├── messages-tuple → 追加到消息列表
    ├── custom → 根据子类型处理（工具调用、进度等）
    └── end → 标记 Run 完成
    │
    ▼ React 状态更新 → UI 渲染
```

前端区分流式 token（逐字渲染）和完整消息（整体渲染）。

## 问题 6：请求中断怎么处理？

用户发送新消息会触发中断：

```
用户正在等待 Agent 回复
    │
    ▼ 用户发送新消息
    │
    ▼ multitask 策略决定
    ├── interrupt → 取消当前 Run，执行新请求
    └── reject → 返回 409，提示用户等待
```

中断时 `dangling_tool_call_middleware` 修复缺失的 ToolMessage（参见 [012-LLM错误处理与重试](012-LLM错误处理与重试.md)）。

## 问题 7：RunJournal 是什么？

`RunJournal` 是 LangChain 的 CallbackHandler，独立于 StreamBridge 记录事件：

| 组件 | 目的 | 存储 |
|------|------|------|
| StreamBridge | 实时推送到前端 | 内存（256 条） |
| RunJournal | 持久化事件记录 | 数据库 |
| Checkpointer | LangGraph 状态快照 | 数据库 |

三者独立运行，互不依赖。StreamBridge 断了不影响持久化。

---

## 问题 8：去重怎么处理？

多路径可能产生重复事件。去重机制：

```python
# 三层去重集合
seen_ids: set        # StreamBridge 已发布
streamed_ids: set    # SSE 已推送
counted_usage_ids: set  # token 统计已计入
```

同一事件不会被重复推送或重复计费。

---

## 数据流概览

```
POST /api/threads/{id}/runs
    │
    ▼ Gateway（认证、CSRF）
    │
    ▼ RunManager（获取锁、创建 Run）
    │
    ▼ Agent 构建（装配中间件）
    │
    ▼ LangGraph 执行循环
    │   ├── before_model: 上下文注入、压缩、todo
    │   ├── LLM 调用（错误重试）
    │   ├── after_model: 工具过滤
    │   ├── before_tool: 护栏检查
    │   ├── 工具执行（沙箱隔离）
    │   └── after_tool: 错误处理、循环检测
    │
    ├── StreamBridge → SSE → 前端实时渲染
    ├── RunJournal → 数据库持久化
    └── Checkpointer → 状态快照
    │
    ▼ Run 完成，释放锁
```
