# 工具同步机制

**问题**: LangGraph 的 Agent 运行在异步事件循环中，但部分工具（MCP 客户端、旧版同步库）是同步的。需要让同步工具在异步环境中正常工作，反之亦然。

---

## 问题 1：为什么会有同步/异步问题？

LangGraph Agent 本身是异步的（`async def`），但：

- **MCP 工具**通过 stdio/sse 通信，底层是同步阻塞 I/O
- **第三方工具**可能是同步函数，不能直接 `await`
- **ACP Agent** 是异步的，但被同步代码调用时需要桥接

直接在异步循环中调用同步阻塞函数会**冻结整个事件循环**。

---

## 问题 2：同步包装器怎么工作？

`make_sync_tool_wrapper` 创建一个同步函数，内部桥接到异步实现：

```python
def make_sync_tool_wrapper(coro, tool_name):
    def sync_wrapper(*args, **kwargs):
        try:
            loop = asyncio.get_running_loop()
            # 事件循环已在运行 → 提交到线程池
            context = contextvars.copy_context()
            future = _SYNC_TOOL_EXECUTOR.submit(
                context.run,
                lambda: asyncio.run(coro(*args, **kwargs))
            )
            return future.result()
        except RuntimeError:
            # 无事件循环 → 直接运行
            return asyncio.run(coro(*args, **kwargs))

    return sync_wrapper
```

两种情况：
1. **事件循环已运行**（常见）：在新线程中启动独立的 `asyncio.run`，避免嵌套事件循环冲突
2. **无事件循环**：直接 `asyncio.run`

---

## 问题 3：为什么用 ThreadPoolExecutor？

```python
_SYNC_TOOL_EXECUTOR = concurrent.futures.ThreadPoolExecutor(
    max_workers=10,
    thread_name_prefix="tool-sync"
)
```

- 线程池隔离同步调用，不阻塞主事件循环
- `max_workers=10` 限制并发数，防止资源耗尽
- `thread_name_prefix` 方便调试时识别工具线程

---

## 问题 4：上下文变量（ContextVars）怎么传递？

同步和异步之间的线程切换会丢失上下文变量（如 `RunnableConfig`）。解决方案：

```python
context = contextvars.copy_context()
future = _SYNC_TOOL_EXECUTOR.submit(
    context.run,  # 在复制的上下文中执行
    lambda: asyncio.run(coro(*args, **kwargs))
)
```

`copy_context()` 捕获当前上下文快照，`context.run()` 在新线程中恢复。这样 `RunnableConfig`、请求级变量等都能正确传递。

---

## 问题 5：异步包装器怎么工作？

反过来，异步 Agent 调用同步工具时：

```python
async def async_wrapper(*args, **kwargs):
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(
        None,  # 使用默认线程池
        functools.partial(sync_func, *args, **kwargs)
    )
    return result
```

`run_in_executor` 将同步函数放到线程池执行，异步等待结果。

---

## 问题 6：会有死锁吗？

潜在风险场景：

```
线程池 10 个线程全部被同步工具占用
    │
    ▼ 新的工具调用需要线程池
    │
    ▼ 等待空闲线程
    │
    ▼ 但已占用的线程在等异步结果
    │
    ▼ 死锁
```

防护措施：
- `max_workers=10` 是经验值，通常足够
- 子 Agent 有超时机制，不会无限占用
- 关键路径（如 MCP 心跳）使用独立线程池

---

## 问题 7：延迟工具注册（Deferred Tool Registry）是什么？

MCP 工具在 Agent 启动时可能还没准备好。`deferred_tool_registry` 延迟到首次使用时才加载：

```
Agent 启动 → 注册工具名（占位）
    │
    ▼ Agent 首次调用该工具
    │
    ▼ deferred_tool_registry 触发加载
    │
    ▼ 从 MCP 服务器获取工具定义
    │
    ▼ 替换占位符为真实工具
```

这让 Agent 启动更快，不必等待所有 MCP 服务器连接完成。

---

## 问题 8：如何调试同步/异步问题？

关键线索：

- **线程名**：`tool-sync-*` 前缀标识同步包装器的线程
- **超时**：如果工具调用长时间无响应，可能是线程池耗尽
- **RuntimeError: This event loop is already running**：嵌套事件循环，需要检查是否在异步上下文中直接调用了 `asyncio.run()`

---

## 数据流概览

```
异步 Agent 调用同步工具
    │
    ▼ sync_wrapper 拦截
    │
    ▼ copy_context() 保存上下文
    │
    ▼ 提交到 ThreadPoolExecutor
    │
    ▼ 新线程中 asyncio.run(coro)
    │
    ▼ 异步工具执行
    │
    ▼ 结果通过 Future 返回主线程
    │
    ▼ Agent 继续执行
```
