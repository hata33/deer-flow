# 运行时并发控制

**问题**: 多个用户（或同一用户的多次请求）可能同时对同一个 Thread 发起 Run，导致状态冲突、数据覆盖或不可预期的行为。

---

## 问题 1：为什么同一 Thread 不能并行执行？

LangGraph 的状态存储在检查点（Checkpoint）中，检查点是线性的链式结构：

```
Checkpoint 1 → Checkpoint 2 → Checkpoint 3
                                      ↑ 当前状态
```

如果两个 Run 并行写入：
- Run A 从 Checkpoint 3 开始 → 生成 Checkpoint 4a
- Run B 从 Checkpoint 3 开始 → 生成 Checkpoint 4b

状态分叉了。后续读取时不知道该用 4a 还是 4b。

---

## 问题 2：并发策略有哪些？

三种策略，按冲突处理方式区分：

| 策略 | 新请求来了 | 当前 Run | 适用场景 |
|------|-----------|---------|---------|
| `reject` | 返回 409 Conflict | 继续执行 | 不允许打断（长任务） |
| `interrupt` | 新请求执行 | 被取消 | 用户改主意了 |
| `rollback` | 新请求执行 | 被取消并回滚 | 需要干净状态 |

```yaml
runtime:
  multitask_strategy: "interrupt"  # 默认值
```

---

## 问题 3：reject 策略怎么工作？

最简单——直接拒绝：

```
Thread 123 正在执行 Run A
    │
    ▼ 新请求到达（Run B）
    │
    ▼ lock_manager 检查锁
    │
    ▼ Thread 已被锁定
    │
    ▼ 返回 409 Conflict
    {"error": "Thread 123 is busy", "current_run": "run_a_id"}
```

前端收到 409 后提示用户"当前有任务正在执行"。

---

## 问题 4：interrupt 策略怎么工作？

取消当前 Run，但保留已生成的检查点：

```
Thread 123 正在执行 Run A（已到第 5 步）
    │
    ▼ 新请求到达（Run B）
    │
    ▼ 取消 Run A
    │
    ▼ Run A 的检查点保留（Checkpoint 1-5 都在）
    │
    ▼ Run B 从 Checkpoint 5 开始执行
```

Run B 基于 Run A 的最新检查点继续——不是从头开始。

---

## 问题 5：rollback 策略怎么工作？

取消当前 Run，**回滚到 Run 开始之前**的状态：

```
Thread 123 状态 = Checkpoint 10
    │
    ▼ Run A 开始，保存 pre_run_checkpoint = Checkpoint 10
    │
    ▼ Run A 执行了 3 步 → Checkpoint 11, 12, 13
    │
    ▼ 新请求到达（Run B）
    │
    ▼ 取消 Run A
    │
    ▼ 回滚到 pre_run_checkpoint (10)
    │
    ▼ Run B 从 Checkpoint 10 开始（Run A 的痕迹全消除）
```

适用于需要"干净重试"的场景。

---

## 问题 6：锁是怎么管理的？

`lock_manager` 为每个 Thread 维护一个异步锁：

```python
class LockManager:
    def __init__(self):
        self._thread_locks = defaultdict(asyncio.Lock)

    async def acquire(self, thread_id):
        lock = self._thread_locks[thread_id]
        await lock.acquire()
        return lock
```

锁在 Run 开始时获取，结束时释放。不同的 multitask 策略在获取锁失败时采取不同行动。

---

## 问题 7：Run 的完整生命周期是什么？

```
请求到达
    │
    ▼ 获取 Thread 锁
    │  ┌─ reject: 锁被占 → 返回 409
    │  ├─ interrupt: 取消旧 Run → 获取锁
    │  └─ rollback: 取消旧 Run + 回滚 → 获取锁
    ▼
保存 pre_run_checkpoint
    │
    ▼ 创建 Run 实例
    │
    ▼ 执行 Agent 循环
    │   ├── before_model 钩子
    │   ├── LLM 调用
    │   ├── after_model 钩子
    │   ├── before_tool 钩子
    │   ├── 工具执行
    │   └── after_tool 钩子
    │
    ▼ Run 完成
    │
    ▼ 释放 Thread 锁
```

---

## 问题 8：事件流（Event Streaming）怎么工作？

Run 执行过程中通过 SSE（Server-Sent Events）实时推送事件：

```
Run 开始 → event: run_started
LLM 输出 → event: token (流式)
工具调用 → event: tool_call
工具结果 → event: tool_result
Run 结束 → event: run_completed
```

前端通过 `EventSource` 监听，实时渲染 Agent 的思考和行动过程。

---

## 问题 9：回滚后之前的对话还在吗？

在。回滚只影响 LangGraph 检查点，不删除消息记录。用户在前端仍能看到之前的对话，只是 Agent 的"工作记忆"回到了 Run 开始前的状态。

---

## 数据流概览

```
用户请求（Thread 123, "帮我重构代码"）
    │
    ▼ lock_manager.acquire("123")
    │
    ├── 策略=reject + 锁被占 → 409
    │
    └── 获取锁成功
        │
        ▼ 保存 pre_run_checkpoint
        │
        ▼ Agent 执行（事件通过 SSE 推送）
        │
        ▼ 执行完成
        │
        ▼ 释放锁
```
