# 工具调用失败处理

**问题**: Agent 执行工具时可能遇到各种失败——权限拒绝、参数错误、超时、服务不可用。如果每次失败都抛异常终止 Run，Agent 根本无法完成复杂任务。

---

## 问题 1：工具调用经过几层拦截？

7 层中间件依次检查，每层都可能拦截并返回错误消息：

```
Agent 发起 tool_call: bash("npm deploy")
    │
    ▼ ① SandboxAuditMiddleware
    检查命令安全性 → 高危命令直接 block
    │
    ▼ ② DeferredToolFilterMiddleware
    检查工具是否已加载 → 未加载则提示先调 tool_search
    │
    ▼ ③ GuardrailMiddleware
    检查工具权限策略 → 不允许则 denied
    │
    ▼ ④ ToolErrorHandlingMiddleware
    执行工具，捕获所有异常 → 转为 ToolMessage(status="error")
    │
    ▼ 正常返回结果
```

---

## 问题 2：安全审计拦截什么？

`SandboxAuditMiddleware` 对 bash 命令做风险评估：

```python
command, _, verdict, reject_reason = self._pre_process(request)

if verdict == "block":
    # 高危命令（如 rm -rf /, dd, mkfs）
    return ToolMessage(
        content=f"安全违规: 命令被阻止。原因: {reject_reason}",
        status="error"
    )

if verdict == "warn":
    # 中危命令（如 curl 外部 URL）→ 允许但追加警告
    result = handler(request)
    return self._append_warn_to_result(result, command)
```

三级处理：**block**（阻止）、**warn**（警告但允许）、**pass**（直接放行）。

## 问题 3：护栏拒绝后 Agent 能做什么？

`GuardrailMiddleware` 拒绝时返回明确原因：

```python
if not decision.allow:
    return ToolMessage(
        content=f"Guardrail denied: tool '{name}' was blocked ({code}). "
                f"Reason: {reason}. Choose an alternative approach.",
        status="error"
    )
```

Agent 看到拒绝原因后可以：
1. **换工具** — 被 `rm` 拒绝 → 改用 `mv` 移到回收区
2. **改参数** — bash 被拒绝 → 看看是否可以用 `write_file` 代替
3. **请求用户** — 不理解为什么被拒 → `ask_clarification`

护栏评估本身出错时，按 `fail_closed` 配置决定：

```python
except Exception:
    if self.fail_closed:
        # 生产环境：评估出错默认拒绝
        decision = GuardrailDecision(allow=False)
    else:
        # 开发环境：评估出错放行
        return handler(request)
```

## 问题 4：工具执行异常怎么处理？

`ToolErrorHandlingMiddleware` 捕获所有异常，转为错误消息：

```python
try:
    return handler(request)
except GraphBubbleUp:
    raise  # LangGraph 控制流信号，不拦截
except Exception as exc:
    detail = str(exc).strip() or exc.__class__.__name__
    if len(detail) > 500:
        detail = detail[:497] + "..."
    return ToolMessage(
        content=f"Error: Tool '{name}' failed with {exc_type}: {detail}. "
                f"Continue with available context, or choose an alternative tool.",
        tool_call_id=call_id,
        status="error"
    )
```

关键设计点：

| 设计 | 原因 |
|------|------|
| 捕获所有异常（除 GraphBubbleUp） | 保证 Run 不崩溃 |
| 错误消息截断 500 字符 | 防止超长堆栈泄露信息 |
| 保留异常类型名 | Agent 能理解错误类别（TimeoutError vs PermissionError） |
| 返回 ToolMessage 而非抛异常 | 对话继续，Agent 可自主恢复 |

## 问题 5：常见工具错误类型及恢复策略？

| 错误类型 | 触发场景 | Agent 典型恢复 |
|---------|---------|---------------|
| `FileNotFoundError` | 文件不存在 | 先 `ls` 看目录结构，再创建 |
| `PermissionError` | 沙箱只读区写入 | 换到可写区域 |
| `TimeoutExpired` | 命令超过 600 秒 | 拆分为多个短命令 |
| `JSONDecodeError` | 工具输出不是有效 JSON | 用 `bash` 直接处理原始输出 |
| `RateLimitError` | API 调用限流 | 等待后重试或换方案 |
| `ConnectionError` | 外部服务不可达 | 用缓存数据或跳过 |
| `ValueError` | 参数格式错误 | 修正参数后重试 |

## 问题 6：超时怎么处理？

bash 命令硬超时 600 秒：

```python
# local_sandbox.py
result = subprocess.run(
    args,
    timeout=600,  # 10 分钟
    capture_output=True,
    text=True
)
```

超时触发 `subprocess.TimeoutExpired` → 被 `ToolErrorHandlingMiddleware` 捕获 → Agent 收到错误消息。

Agent 的典型应对：
1. 把长任务拆成多步
2. 加 `timeout` 参数到命令本身（如 `timeout 60 npm test`）
3. 后台运行 + 轮询结果

## 问题 7：延迟工具未加载怎么办？

MCP 工具可能还没准备好。`DeferredToolFilterMiddleware` 检查：

```python
def _blocked_tool_message(self, request):
    if not registry.contains(tool_name):
        return None  # 不认识的工具，不处理

    return ToolMessage(
        content=f"Error: Tool '{name}' is deferred. "
                f"Call tool_search first to expose this tool.",
        status="error"
    )
```

Agent 收到后先调用 `tool_search` 发现工具，再重新调用。

## 问题 8：用户中断导致工具结果丢失？

用户在工具执行期间发新消息 → 工具结果没来得及生成 → LangGraph 要求每个 `tool_call` 都有对应 `ToolMessage`。

`DanglingToolCallMiddleware` 在下次 `wrap_model_call` 时自动修复：

```python
# 检查消息列表中是否有 tool_call 缺少对应 ToolMessage
for msg in messages:
    if isinstance(msg, AIMessage) and msg.tool_calls:
        for call in msg.tool_calls:
            if call["id"] not in seen_tool_message_ids:
                # 补一个 ToolMessage
                patched.append(ToolMessage(
                    content="[Tool call was interrupted and did not return a result.]",
                    tool_call_id=call["id"],
                    status="error"
                ))
```

**不需要人工干预**——系统自动修复数据完整性问题。

---

## 问题 9：工具会自动重试吗？

**不会**。工具失败后由 Agent 自主决定是否重试。原因：

| 考量 | 说明 |
|------|------|
| Agent 有上下文 | 它知道上次为什么失败，能智能调整 |
| 不是所有失败都该重试 | 参数错误重试没用，限流重试才有意义 |
| 避免隐藏成本 | 自动重试可能浪费大量 token |

对比 LLM 调用有自动重试（因为 LLM API 失败通常是暂时的，且重试成本可预测）。

## 问题 10：完整的错误恢复流程？

```
工具调用失败
    │
    ▼ 返回 ToolMessage(status="error")
    │
    ▼ Agent 收到错误消息
    │
    ▼ Agent 分析错误原因
    │
    ├── 可恢复（如参数错误）
    │   └── 修正参数 → 重新调用同一工具
    │
    ├── 可替代（如工具被拒绝）
    │   └── 选择其他工具完成相同任务
    │
    ├── 需要信息（如文件不存在）
    │   └── 先探索（ls/read_file）→ 再执行
    │
    ├── 需要用户（如权限不足）
    │   └── ask_clarification → 等用户确认
    │
    └── 无法恢复
        └── 向用户报告失败原因
```

## 数据流概览

```
Agent 发起 tool_call: bash("npm deploy")
    │
    ▼ SandboxAudit → verdict=pass
    │
    ▼ DeferredToolFilter → 已加载
    │
    ▼ Guardrail → allowed
    │
    ▼ ToolErrorHandling → 执行
    │
    ├── 成功 → ToolMessage(content=结果)
    │
    └── 异常 → ToolMessage(
    │       content="Error: Tool 'bash' failed with TimeoutExpired: ...",
    │       status="error"
    │   )
    │
    ▼ Agent 看到错误
    │
    ▼ 分析 + 决策 → 重试/换工具/请求帮助
```
