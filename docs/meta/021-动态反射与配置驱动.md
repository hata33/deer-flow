# 动态反射与配置驱动

**问题**: DeerFlow 需要支持可插拔的 Provider、工具、沙箱实现等，但硬编码 `import` 会导致耦合度高、扩展困难。通过反射动态加载，让配置决定代码行为。

---

## 问题 1：什么是反射加载？

通过字符串路径动态导入 Python 类/函数：

```yaml
# 配置
guardrails:
  provider: "deerflow.guardrails.builtin:AllowlistProvider"
```

```python
# 反射加载
from deerflow.reflection.resolvers import resolve

provider_class = resolve("deerflow.guardrails.builtin:AllowlistProvider")
# 等同于: from deerflow.guardrails.builtin import AllowlistProvider
```

冒号分隔模块路径和变量名。

---

## 问题 2：哪些地方用了反射？

| 场景 | 配置路径 | 加载的类 |
|------|---------|---------|
| 模型 Provider | `models.providers.*.provider_class` | BaseChatModel 子类 |
| 护栏 Provider | `guardrails.provider` | GuardrailProvider |
| 沙箱 Provider | `sandbox.provider_class` | SandboxProvider |
| 工具加载 | 自定义工具配置 | BaseTool 子类 |
| 记忆存储 | `memory.storage_class` | MemoryStorage |
| 追踪 Provider | `tracing.providers.*` | CallbackHandler |
| MCP 拦截器 | `mcpInterceptors` | 拦截器函数 |

几乎每个可插拔组件都通过反射加载。

---

## 问题 3：模块不存在怎么办？

反射加载失败时给出友好的安装提示：

```python
MODULE_TO_PACKAGE_HINTS = {
    "langchain_anthropic": "pip install langchain-anthropic",
    "langchain_openai": "pip install langchain-openai",
    "langchain_community": "pip install langchain-community",
    ...
}

# 缺少模块时
try:
    module = importlib.import_module(module_path)
except ImportError:
    hint = MODULE_TO_PACKAGE_HINTS.get(module_path, "")
    raise ImportError(
        f"Module '{module_path}' not found. "
        f"Install it with: {hint}"
    )
```

用户不再看到晦涩的 `ModuleNotFoundError`，而是直接知道该装什么。

## 问题 4：类型安全怎么保证？

反射加载后做类型检查：

```python
def resolve(path: str, expected_type: type = None):
    obj = _load_object(path)

    if expected_type:
        if isinstance(expected_type, type):
            if not isinstance(obj, expected_type):
                raise TypeError(
                    f"{path} is {type(obj)}, expected {expected_type}"
                )
        elif isinstance(expected_type, tuple):  # Union type
            if not isinstance(obj, expected_type):
                raise TypeError(...)

    return obj
```

配置错误在启动时就被发现，不会等到运行时才报错。

## 问题 5：配置驱动的特性系统是什么？

`RuntimeFeatures` 数据类控制 Agent 的行为：

```python
@dataclass
class RuntimeFeatures:
    summarization: bool | SummarizationMiddleware = True
    loop_detection: bool | LoopDetectionMiddleware = True
    memory: bool | MemoryMiddleware = True
    guardrails: bool | GuardrailsMiddleware = True
    ...
```

每个特性有三种值：
- `True`：启用，使用默认中间件
- `False`：禁用
- `中间件实例`：启用，使用自定义中间件

```yaml
features:
  summarization: true
  loop_detection: false     # 禁用循环检测
  memory: true
```

## 问题 6：中间件的位置怎么控制？

`@Next` 和 `@Prev` 装饰器声明式指定中间件执行顺序：

```python
@Next(SummarizationMiddleware)  # 必须在压缩之后
class TodoMiddleware:
    ...

@Prev(LoopDetectionMiddleware)  # 必须在循环检测之前
class ToolErrorHandlingMiddleware:
    ...
```

为什么声明式优于硬编码顺序：
- 新增中间件不需要修改已有代码
- 顺序关系是语义约束，不是偶然排列
- 冲突时启动报错，不会默默执行错误顺序

---

## 问题 7：配置热更新怎么实现？

部分配置支持运行时更新，不需要重启：

| 配置 | 热更新 | 机制 |
|------|--------|------|
| MCP 工具 | 支持 | 文件修改时间检测 |
| 技能 | 支持 | 文件修改时间检测 |
| 模型 Provider | 不支持 | 需要重启 |
| 护栏策略 | 不支持 | 需要重启 |
| 通道配置 | 不支持 | 需要重启 |

MCP 和技能的缓存通过 `_is_cache_stale()` 方法检查配置文件变化。

---

## 问题 8：如何添加自定义组件？

四步：

```
1. 实现接口
   class MyGuardrailProvider(GuardrailProvider):
       def evaluate(self, request): ...

2. 打包安装
   pip install my-guardrail

3. 配置引用
   guardrails:
     provider: "my_guardrail:MyGuardrailProvider"

4. 重启生效
   反射加载自动找到你的类
```

不需要修改 DeerFlow 的任何源代码。

## 数据流概览

```
config.yaml 加载
    │
    ▼ 解析配置项
    │
    ▼ 遇到 "module.path:ClassName"
    │
    ▼ reflection.resolvers.resolve()
    │
    ├── importlib.import_module() → 导入模块
    ├── getattr() → 获取类/函数
    ├── 类型检查 → 确保符合预期类型
    │
    ▼ 返回类引用
    │
    ▼ 实例化并注入系统
```
