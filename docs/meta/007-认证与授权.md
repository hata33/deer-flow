# 认证与授权

**问题**: DeerFlow 需要保护 API 端点不被未授权访问，同时支持多种认证方式（本地密码、OAuth、内部服务），还要防御 CSRF、会话劫持等 Web 攻击。

---

## 问题 1：支持哪些认证方式？

三种认证策略并行：

| 方式 | 适用场景 | 实现 |
|------|---------|------|
| 本地密码 | 自托管/开发环境 | `local_provider` — bcrypt 哈希 |
| OAuth 2.0 | 社交登录 | GitHub 等 OAuth Provider |
| 内部认证 | 微服务间调用 | `internal_auth` — 共享密钥 |

```yaml
auth:
  jwt_secret: "${AUTH_JWT_SECRET}"
  token_expiry_days: 7
  oauth:
    github:
      client_id: "${GITHUB_CLIENT_ID}"
      client_secret: "${GITHUB_CLIENT_SECRET}"
```

---

## 问题 2：JWT Token 的结构是什么？

```json
{
    "sub": "<user_uuid>",
    "exp": "<expiry_time>",
    "iat": "<issued_at>",
    "ver": 1
}
```

关键字段 `ver`（版本号）：用户修改密码时递增，旧 token 的 `ver` 不匹配则失效，实现**密码修改即全局登出**。

---

## 问题 3：密码怎么存储？

两代哈希方案：

| 版本 | 格式 | 说明 |
|------|------|------|
| v1 | `$dfv1$<bcrypt>` | 纯 bcrypt |
| v2 | `$dfv2$<bcrypt>` | SHA-256 预哈希 + bcrypt |

v2 的 SHA-256 预哈希解决了超长密码导致的 bcrypt 截断问题。系统自动识别版本，登录时用对应算法验证，验证后自动升级到 v2。

---

## 问题 4：CSRF 怎么防护？

双重提交 Cookie 模式：

```
1. 服务端设置 HttpOnly Cookie: csrf_token=<random>
2. 前端读取 Cookie，在请求头附加: X-CSRF-Token: <same_value>
3. 服务端比对 Cookie 和 Header 是否一致
```

```python
CSRF_COOKIE_NAME = "csrf_token"
CSRF_HEADER_NAME = "X-CSRF-Token"
```

为什么有效：恶意网站无法读取目标的 Cookie，自然无法在 Header 中提交正确的值。

---

## 问题 5：JWT Secret 怎么管理？

优先级链：

```
1. 环境变量 AUTH_JWT_SECRET（推荐）
    ↓ 未设置
2. .jwt_secret 文件（开发环境）
    ↓ 不存在
3. 启动时自动生成（重启后 token 失效）
```

生产环境必须通过环境变量设置，避免 secret 泄露到文件系统。

---

## 问题 6：内部认证（服务间）怎么工作？

`internal_auth` 用于后端微服务之间的调用，不走 JWT：

```python
# 请求头
X-Internal-Token: <shared_secret>
```

内部请求跳过用户认证逻辑，直接获得系统级权限。`internal_auth` 中间件在认证链中最先执行，匹配则短路，不匹配则继续走外部认证流程。

---

## 问题 7：Token 过期后怎么办？

JWT 过期后前端引导用户重新登录。目前没有 refresh token 机制——这是有意为之：

- **简化实现**：不需要 refresh token 的存储和轮换逻辑
- **安全考量**：减少 token 被窃取后的攻击窗口
- **适用场景**：DeerFlow 定位为开发工具而非面向公众的 SaaS

如需更长的会话，调整 `token_expiry_days`。

---

## 问题 8：认证中间件的执行顺序？

```
CSRF 中间件（验证 CSRF Token）
    │
    ▼
认证中间件（JWT / OAuth / Internal）
    │
    ▼
路由处理器（已确认身份的请求）
```

CSRF 在认证之前执行，确保伪造请求在认证前就被拒绝。

---

## 数据流概览

```
请求到达
    │
    ▼ CSRF 中间件
比对 Cookie 和 Header
    │
    ▼ 通过
认证中间件
    ├── X-Internal-Token → internal_auth → 系统级权限
    ├── Authorization: Bearer → JWT 验证 → 用户级权限
    └── OAuth 回调 → 交换 token → 用户级权限
    │
    ▼ 认证成功
请求注入 user_id，转发到路由处理器
```
