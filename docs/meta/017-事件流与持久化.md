# 事件流与持久化

**问题**: Agent 执行过程产生大量事件（LLM 输出、工具调用、状态变更），需要同时满足三个需求：实时推送到前端、持久化到数据库、状态快照恢复。这三件事不能混为一谈。

---

## 问题 1：三个独立的机制是什么？

| 机制 | 组件 | 目的 | 存储 |
|------|------|------|------|
| 实时推送 | StreamBridge | 前端流式渲染 | 内存（256 条） |
| 事件记录 | RunJournal | 审计、回放、调试 | 数据库 |
| 状态快照 | Checkpointer | 恢复执行 | 数据库 |

三者互不依赖——StreamBridge 断了不影响持久化，数据库挂了不影响实时推送。

---

## 问题 2：StreamBridge 的工作原理？

发布-订阅模式：

```
Agent 执行
    │
    ▼ publish(event)
StreamBridge（内存队列）
    │
    ▼ consume()
SSE 连接 → 前端
```

```python
class MemoryStreamBridge:
    def __init__(self):
        self._events = deque(maxlen=256)  # 保留最近 256 条
        self._subscribers = []

    async def publish(self, event):
        self._events.append(event)
        for sub in self._subscribers:
            await sub.put(event)
```

事件 ID 格式：`{timestamp}-{sequence}`，保证有序且唯一。

---

## 问题 3：RunJournal 记录什么？

`RunJournal` 是 LangChain CallbackHandler，记录 LLM 调用的详细信息：

| 事件 | 内容 | 用途 |
|------|------|------|
| `on_llm_start` | 模型名、输入 token 数 | 调试 |
| `on_llm_end` | 输出 token 数、耗时 | 成本分析 |
| `on_tool_start` | 工具名、参数 | 审计 |
| `on_tool_end` | 工具输出 | 回放 |
| `on_chain_error` | 错误信息 | 故障排查 |

## 问题 4：Run 持久化存什么？

`RunRow` 存储每次 Run 的元数据：

```python
class RunRow:
    id: str              # Run ID
    thread_id: str       # 所属 Thread
    status: str          # pending/running/success/error/timeout/interrupted
    input_tokens: int    # 输入 token 总量
    output_tokens: int   # 输出 token 总量
    subagent_tokens: int # 子 Agent token
    started_at: datetime
    finished_at: datetime
    error: str | None
    metadata: dict       # 扩展元数据
```

Token 统计包含子 Agent 的消耗——用户看到的是总量。

---

## 问题 5：RunEvent 持久化存什么？

`RunEventRow` 存储有序事件流，用于回放和分页：

```python
class RunEventRow:
    id: int              # 自增主键（非 UUID，保证顺序）
    thread_id: str
    run_id: str
    seq: int             # 事件序号（每 Thread 独立）
    category: str        # message / trace / lifecycle
    event_type: str      # 具体事件类型
    data: dict           # 事件数据（JSON）
    timestamp: datetime
```

唯一约束：`(thread_id, seq)` — 保证事件顺序。

---

## 问题 6：为什么要自增主键而非 UUID？

事件需要**严格有序**。自增主键天然有序，查询时直接按 PK 排序。UUID 需要额外的时间戳排序，且无法保证精确顺序。

```
自增 ID: 1, 2, 3, 4, 5 → 顺序天然正确
UUID: a3f2..., 7b1c..., e9d4... → 需要额外排序
```

---

## 问题 7：去重怎么处理？

三个机制可能产生重复事件。去重集合：

```python
# 发布时去重
seen_ids: set          # 已发布到 StreamBridge

# 推送时去重
streamed_ids: set      # 已通过 SSE 推送

# 计费时去重
counted_usage_ids: set # 已计入 token 统计
```

同一事件不会被重复推送、重复记录或重复计费。

## 问题 8：Run 状态有哪些？

```
pending → running → success
                 → error
                 → timeout
                 → interrupted（用户中断）
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `pending` | 已创建，等待执行 | 请求到达 |
| `running` | 正在执行 | 获取到锁 |
| `success` | 正常完成 | Agent 输出最终结果 |
| `error` | 执行出错 | 未捕获异常 |
| `timeout` | 超时 | 超过 `timeout_seconds` |
| `interrupted` | 用户中断 | 新请求打断 |

---

## 数据流概览

```
Agent 执行事件
    │
    ├── StreamBridge → 内存队列 → SSE → 前端实时渲染
    │
    ├── RunJournal → RunEventRow → 数据库（事件流持久化）
    │
    └── Checkpointer → LangGraph 状态快照 → 数据库（状态恢复）
    │
    ▼ Run 完成
RunRow 更新（状态、token、耗时）
```
