# MCP工具与缓存

**问题**: MCP（Model Context Protocol）服务器提供的工具初始化成本高（需启动进程、建立连接、获取工具列表），频繁重新初始化会拖慢 Agent 启动速度和响应时间。

---

## 问题 1：MCP 是什么？

MCP（Model Context Protocol）是一个开放协议，让 LLM 应用能连接外部工具服务器。每个 MCP 服务器提供一组工具：

```
DeerFlow Agent
    │
    ├── MCP Server: filesystem（文件操作工具）
    ├── MCP Server: brave-search（搜索工具）
    └── MCP Server: github（GitHub 工具）
```

每个服务器是独立进程，通过 stdio 或 SSE 通信。

---

## 问题 2：MCP 服务器怎么配置？

在 `extensions_config.json` 中声明：

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path"],
      "env": {"NODE_ENV": "production"}
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": {"BRAVE_API_KEY": "${BRAVE_API_KEY}"}
    }
  }
}
```

配置支持环境变量替换（`${VAR_NAME}`），避免硬编码密钥。

---

## 问题 3：缓存怎么工作？

`MCPToolsCache` 避免每次请求都重新连接 MCP 服务器：

```python
class MCPToolsCache:
    def __init__(self):
        self._tools: list | None = None
        self._cache_initialized = False
        self._initialization_lock = asyncio.Lock()
        self._config_mtime: float | None = None
```

流程：
1. 首次请求 → 连接所有 MCP 服务器 → 获取工具列表 → 缓存
2. 后续请求 → 直接用缓存（毫秒级）
3. 配置文件变更 → 下次请求时重新初始化

---

## 问题 4：怎么知道缓存过期了？

通过配置文件的修改时间判断：

```python
def _is_cache_stale(self):
    current_mtime = self._get_config_mtime()
    if self._config_mtime is None:
        return True
    return current_mtime > self._config_mtime
```

配置文件被编辑后，修改时间变化，下次访问触发重新加载。这是**被动检测**——不主动监听文件变化，而是在使用时检查。

---

## 问题 5：多个 Run 同时初始化缓存怎么办？

`asyncio.Lock` 保证只有一个协程执行初始化：

```python
async def initialize_mcp_tools(self):
    async with self._initialization_lock:  # 互斥锁
        if self._cache_initialized and not self._is_cache_stale():
            return self._tools  # 已初始化且未过期

        self._tools = await self._load_mcp_tools()  # 只有一个协程执行
        self._cache_initialized = True
        return self._tools
```

其他协程等待锁释放后，看到 `_cache_initialized=True`，直接用缓存。

---

## 问题 6：MCP 连接失败怎么办？

单个服务器失败不影响其他服务器：

```
_load_mcp_tools()
    ├── 连接 filesystem → 成功 → 收集工具
    ├── 连接 brave-search → 超时 → 记录警告，跳过
    └── 连接 github → 成功 → 收集工具
    │
    ▼ 返回 [filesystem工具..., github工具...]
```

Agent 可以正常使用已连接的工具，只是缺少失败服务器的工具。

---

## 问题 7：MCP 工具怎么注册到 Agent？

MCP 工具通过 `tool_assembler` 包装为 LangChain `BaseTool`，注册到工具列表中。同步/异步桥接由 `sync.py` 中的包装器自动处理。

```
MCP Server 工具列表
    │
    ▼ tool_assembler 包装
    │
    ▼ sync.py 处理同步/异步桥接
    │
    ▼ 注册到 Agent 的工具集
    │
    ▼ Agent 可像调用内置工具一样调用
```

---

## 问题 8：OAuth 认证怎么处理？

部分 MCP 服务器需要 OAuth 授权。`oauth.py` 管理 OAuth 流程：

```
1. Agent 首次调用需要 OAuth 的工具
2. 检测到未授权 → 返回授权 URL
3. 用户在浏览器中授权
4. 回调接收授权码 → 交换 access_token
5. token 缓存，后续调用自动附加
```

token 刷新也是自动的——过期时自动重新获取。

---

## 问题 9：MCP 拦截器是什么？

`mcpInterceptors` 允许在工具调用前后插入自定义逻辑：

```json
{
  "mcpInterceptors": [
    "my_company.interceptors:build_logging_interceptor"
  ]
}
```

拦截器可以：
- 记录工具调用日志
- 修改工具参数
- 过滤工具返回结果
- 实现自定义审计

---

## 数据流概览

```
extensions_config.json（MCP 配置）
    │
    ▼ 首次请求
MCPToolsCache.initialize_mcp_tools()
    │
    ▼ asyncio.Lock 互斥
    │
    ▼ 连接 MCP 服务器
    ├── filesystem → 工具列表
    ├── brave-search → 工具列表
    └── github → 工具列表（OAuth）
    │
    ▼ tool_assembler 包装
    │
    ▼ 缓存（直到配置变更）
    │
    ▼ Agent 调用 MCP 工具
    │
    ▼ sync_wrapper 桥接到异步 MCP 调用
    │
    ▼ 结果返回 Agent
```
