# Agentic RAG 方案可行性分析

**约束**: 不修改 DeerFlow 现有架构。所有能力通过已有的扩展机制（社区工具、技能、配置）注入。

---

## 1. 结论

方案**理念正确**，但实现细节存在 6 个问题。修正后可完全在不改架构的前提下落地。

| 维度 | 评估 |
|------|------|
| 核心思路 | Agent 自主决定检索时机和策略 → 正确，DeerFlow 的 ReAct 循环天然支持 |
| 工具注册 | 社区工具 + config.yaml `use:` 字段 → 完全可行 |
| 技能定义 | SKILL.md + `allowed-tools` 白名单 → 完全可行 |
| 中间件 | 需要新增 `rag_context_middleware.py` → **不需要，应删除** |
| 配置扩展 | 新增 `agentic_rag` 配置段 → 需处理，但不强制改 AppConfig |
| 向量存储 | VectorStoreManager 实例管理 → **有 bug，需修正** |

---

## 2. 问题逐条分析

### 问题 1: `rag_context_middleware.py` 与"不改架构"矛盾

**位置**: 方案第 3 节「需要新建的文件」

方案列出了 `rag_context_middleware.py` 作为新中间件。但中间件装配链在 `agents/lead_agent/agent.py:_build_middlewares()` 中是硬编码的顺序序列。加入新中间件**必须修改这个函数**，这违反了"不改架构"的约束。

更关键的是，这个中间件**与方案自身理念矛盾**。Agentic RAG 的核心是让 Agent 自主决定何时检索 — 这正是工具的职责。如果在中间件（`before_model`）阶段自动注入检索结果，就退化成了传统 RAG 的固定 pipeline。

**修正**: 删除 `rag_context_middleware.py`，所有检索能力通过工具暴露。

### 问题 2: Embedding 模型没有基础设施

**位置**: 方案第 7 节 `document_processor.py:462`

```python
from .tools import _get_embeddings  # 这个函数从未定义
embeddings = _get_embeddings()
```

DeerFlow 当前只有 LLM 模型的工厂方法 `create_chat_model()`，**没有 embedding 模型的配置、初始化、API Key 管理**。这不是几行代码的事：

- embedding API Key 可能和 LLM 的不同（OpenAI embedding + Anthropic LLM 很常见）
- 需要独立的 `base_url`、`api_key`、`model_name` 配置
- 向量维度必须和存储后端匹配（`text-embedding-3-small` 是 1536 维）

**修正**: 在 `community/agentic_rag/config.py` 中自行读取配置，不依赖 AppConfig 的结构化字段。`AppConfig` 有 `extra="allow"`（`app_config.py:161`），config.yaml 中加 `agentic_rag:` 不会崩溃，但数据在 `model_extra` 中，需手动读取：

```python
# community/agentic_rag/config.py
from deerflow.config.app_config import get_app_config

def get_rag_config() -> dict:
    app_config = get_app_config()
    return app_config.model_extra.get("agentic_rag", {})
```

### 问题 3: VectorStoreManager 不是单例

**位置**: 方案第 7 节 `document_processor.py:465`

```python
vs = VectorStoreManager()      # 每次工具调用都新建
embeddings = _get_embeddings() # 每次都初始化
```

每次工具调用都重新创建 VectorStoreManager 和 embedding 客户端。Chroma/FAISS 的索引加载很重（大知识库可能数秒），之前 `_stores` 缓存全部丢失。

**修正**: 模块级全局变量 + 懒初始化，复用 `get_app_config()` 单例机制：

```python
# vector_store.py
_manager: VectorStoreManager | None = None

def get_vector_store_manager() -> VectorStoreManager:
    global _manager
    if _manager is None:
        config = get_rag_config()
        _manager = VectorStoreManager(backend=config.get("backend", "chroma"))
    return _manager
```

### 问题 4: `keyword_search` 实现是假的

**位置**: 方案第 6 节 `retriever.py:339-347`

```python
def keyword_search(self, keywords, collection, top_k):
    store = self._vs.get_store(collection, self._embeddings)
    results = store.similarity_search(keywords, k=top_k)  # 还是语义搜索
    return results
```

实际调的还是 `similarity_search`（向量语义搜索），不是关键词匹配。两个"不同"工具底层一样，LLM 还得花 token 决定用哪个，增加决策成本。

Chroma 的关键词搜索应使用 `where_document={"$contains": keyword}`，FAISS 不支持关键词搜索。

**修正**: 两个选项：

- **选项 A**: 正确实现关键词搜索（用 Chroma `where_document`），保留两个工具
- **选项 B**（推荐）: 只保留一个 `knowledge_search` 工具，内部自动尝试语义+精确匹配，减少 Agent 的决策负担

### 问题 5: 工具定位混淆了运行时和管理操作

**位置**: 方案第 4.1 节「工具清单」

6 个工具中，3 个是管理操作：

| 工具 | 实际用途 | 问题 |
|------|---------|------|
| `index_documents` | 索引文档 | embedding 批量调用可能几十秒到分钟级，阻塞 Agent 运行 |
| `list_collections` | 查看知识库状态 | 用户对话中用不到 |
| `get_document_info` | 查看文档元数据 | 运维/管理功能 |

Agent 对话时实际只需要**检索类工具**。`index_documents` 特别危险 — 如果用户上传一个 50 页 PDF 并在对话中说"帮我索引这个文件"，Agent 会调用此工具，整个对话阻塞在 embedding API 调用上。

**修正**:

- 运行时工具只保留检索类：`knowledge_search`（语义）、`exact_search`（精确匹配）
- `index_documents` 拆为离线 CLI（`deer-flow rag index /path/to/docs`）或子代理后台任务
- `list_collections` / `get_document_info` 走管理 API，不走对话工具

### 问题 6: 沙箱路径冲突

**位置**: 方案第 11.3 节

```python
index_documents("/mnt/user-data/uploads/新员工手册.pdf")
```

沙箱使用虚拟路径（`/mnt/user-data/*` → 物理路径翻译），但 RAG 工具不是沙箱工具，拿不到 `sandbox` 实例来做路径翻译。向量存储路径 `.deer-flow/vector_store` 是物理路径，和沙箱隔离模型冲突。

**修正**: 检索工具通过 `state["thread_data"]` 获取物理路径（现有中间件已注入），不走沙箱虚拟路径。或者索引操作完全放在沙箱外。

## 3. 可行且设计合理的部分

| 部分 | 评价 |
|------|------|
| 工具注册方式 | `community/agentic_rag/tools.py` + config.yaml `use:` 字段，完全符合现有社区工具模式 |
| SKILL.md 技能定义 | `allowed-tools` 白名单 + 检索决策框架提示词，是正确的技能用法 |
| Agent 推理流程描述 | 三个场景（简单/多轮/失败恢复）的描述准确，ReAct 循环确实会这样推理 |
| RRF 融合排序 | `Reciprocal Rank Fusion` 是成熟的检索融合算法，实现正确 |
| 混合检索的语义权重 | `semantic_weight: 0.7` 可配置，合理 |

## 5. 修正后的工具清单

| 工具 | 功能 | 理由 |
|------|------|------|
| `knowledge_search` | 语义检索（默认推荐） | 核心能力，Agent 首选 |
| `exact_search` | Chroma `where_document` 精确匹配 | 语义检索不准时的后备 |
| ~~`hybrid_search`~~ | 删除 | Agent 自己决定是否组合两次检索 |
| ~~`keyword_search`~~ | 删除 | 实现是假的，用 `exact_search` 替代 |
| ~~`index_documents`~~ | 拆为离线 CLI | 运行时太重，阻塞对话 |
| ~~`list_collections`~~ | 删除 | 管理操作，不走对话工具 |
| ~~`get_document_info`~~ | 删除 | 管理操作，不走对话工具 |

2 个运行时工具 + 1 个离线索引命令，比原方案的 6 个对话工具更干净。

## 6. 按 Agent 粒度的配置隔离

DeerFlow 现有架构**已经支持**通过 config.yaml 控制不同 Agent 使用不同的工具、技能。无需修改代码。

### 6.1 控制机制

**三个层级的配置隔离，全部通过 config.yaml 驱动：**

#### 层级 1: Tools — 通过 `tool_groups` 过滤

config.yaml 中工具按 `group` 分组注册：

```yaml
tools:
  - name: knowledge_search
    group: rag
    use: deerflow.community.agentic_rag.tools:knowledge_search_tool

  - name: exact_search
    group: rag
    use: deerflow.community.agentic_rag.tools:exact_search_tool

  - name: web_search
    group: web
    use: deerflow.community.ddg_search.tools:web_search_tool
```

`get_available_tools()` 根据 `groups` 参数过滤（`agent.py` 中调用时传入 `agent_config.tool_groups`）。

#### 层级 2: Skills — 通过 `skills` 白名单控制

`skills/custom/agentic-rag/SKILL.md` 定义技能和允许的工具：

```yaml
---
name: agentic-rag
allowed-tools:
  - knowledge_search
  - exact_search
  - web_search
  - read_file
  - ask_clarification
---
```

`apply_prompt_template()` 根据 `agent_config.skills` 过滤技能加载。

#### 层级 3: 子代理 — 独立的 tools/skills/model

```yaml
subagents:
  custom_agents:
    - name: "rag-researcher"
      model: "gpt-4o"
      max_turns: 30
      tools: [knowledge_search, exact_search, web_search, read_file]
      skills: [agentic-rag]
```

### 6.2 不同 Agent 的配置组合

| Agent 类型 | tool_groups | skills | 效果 |
|-----------|-------------|--------|------|
| 默认 lead agent | 不配（全部工具） | 不配（全部技能） | 有 RAG 工具但无 RAG 专用提示词 |
| RAG 专用 agent | `[rag, web]` | `[agentic-rag]` | 只有 RAG 工具 + RAG 决策提示词 |
| 无 RAG 的 agent | `[web, bash]` | `[other-skill]` | 完全没有 RAG 工具 |
| RAG 子代理 | 子代理 config 的 tools 字段 | 子代理 config 的 skills 字段 | 子代理内独立 RAG 能力 |

**关键点**: `tool_groups` 和 `skills` 是**白名单机制** — 只有不配或配 `null` 时才加载全部。一旦配置了具体值，该 Agent 只能使用列出的工具和技能。

### 6.3 自定义 Agent 配置示例

**RAG 专用 Agent**:

```yaml
# .deer-flow/users/{user_id}/agents/rag-assistant/config.yaml
name: rag-assistant
description: "Agentic RAG 知识检索助手"
tool_groups:
  - rag        # knowledge_search, exact_search
  - web        # web_search
  - file:read  # read_file
skills:
  - agentic-rag
model: "gpt-4o"
```

**通用 Agent（有 RAG 但不专精）**:

```yaml
# .deer-flow/users/{user_id}/agents/general/config.yaml
name: general
description: "通用助手"
# 不配 tool_groups → 全部工具（含 RAG）
# 不配 skills → 全部技能（含 agentic-rag）
model: "claude-sonnet-4-6"
```

**无 RAG 的专用 Agent**:

```yaml
# .deer-flow/users/{user_id}/agents/coder/config.yaml
name: coder
description: "代码助手"
tool_groups:
  - bash
  - file
skills:
  - code-review  # 不含 agentic-rag → 无 RAG 能力
```

### 6.4 中间件不可按 Agent 配置

现有架构中 `_build_middlewares()` 不支持按 Agent 类型选择中间件。但 RAG 场景**不需要新中间件** — 检索决策通过工具 + 技能提示词完成，这正是 Agentic RAG 的核心设计。中间件不需要动。

## 7. 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| embedding API 延迟影响对话响应 | 中 | 检索工具返回缓存结果，embedding 预计算 |
| 向量库文件与线程隔离冲突 | 低 | 检索只读，不在线程目录内写向量库 |
| 知识库内容安全（敏感文档泄露） | 中 | collection 按用户/权限隔离，检索工具校验 user_id |
| Chroma 单进程写入锁 | 低 | 索引走离线 CLI，运行时只读 |
| 无 reranker 导致检索质量不够 | 低 | RRF 融合 + Agent 自主判断相关性，可后续加 reranker |
