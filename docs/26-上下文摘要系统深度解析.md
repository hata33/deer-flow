# 【26】上下文摘要系统深度解析

## 1. 模块全局定位

- **所属项目**：deer-flow
- **层级位置**：`backend/packages/harness/deerflow/agents/middlewares/` + `config/summarization_config.py`
- **核心作用**：自动压缩长对话历史，防止上下文超出模型Token限制
- **业务价值**：作为"上下文管理器"，在对话增长时自动总结早期消息，保留近期完整历史，平衡信息完整性与Token消耗
- **设计初衷**：设计用于解决"长对话Token溢出"问题——当对话超过模型max_tokens时，自动压缩历史避免API错误

## 2. 核心设计思想

**触发条件**（支持多条件OR）：
- `messages`：消息数量达到阈值（如50条）
- `tokens`：Token数量达到阈值（如4000）
- `fraction`：达到模型max_tokens的百分比（如0.8=80%）

**保留策略**（压缩后保留）：
- `messages`：保留N条近期消息（如20条）
- `tokens`：保留N个Token（如3000）
- `fraction`：保留max_tokens的百分比（如0.3=30%）

**摘要模型**：
- 默认使用轻量模型节省成本
- 可配置`model_name`指定专用模型
- 支持自定义`summary_prompt`

## 3. 关键配置解读

```python
class SummarizationConfig(BaseModel):
    enabled: bool = False  # 默认禁用
    model_name: str | None = None  # null=轻量模型
    trigger: ContextSize | list[ContextSize] | None = None  # 触发条件
    keep: ContextSize = ContextSize(type="messages", value=20)  # 保留策略
    trim_tokens_to_summarize: int | None = 4000  # 摘要输入Token限制
    summary_prompt: str | None = None  # 自定义摘要提示词
```

**设计考量**：
- **默认禁用**：摘要会丢失细节，默认关闭避免意外数据损失
- **多触发条件**：支持灵活触发策略，适应不同模型与场景
- **独立保留策略**：触发条件与保留策略独立配置，精细控制压缩行为
- **输入Token限制**：`trim_tokens_to_summarize`防止摘要输入过长导致二次溢出

## 4. 工作流程

1. **检测触发条件**：每次对话后检查是否满足任一触发条件
2. **选择摘要范围**：保留近期消息（按keep策略），早期消息送入摘要
3. **调用摘要模型**：使用配置的模型生成早期消息摘要
4. **替换历史消息**：用摘要消息替换原始早期消息
5. **保留近期完整**：keep策略指定数量的近期消息保持原样

## 5. 最佳实践

1. **渐进启用**：开发阶段禁用，生产环境根据Token使用情况启用
2. **保留策略设置**：保留足够近期消息（如20条）确保上下文连续性
3. **轻量模型**：摘要任务相对简单，使用快速模型节省延迟
4. **监控调整**：观察摘要质量与Token消耗，动态调整触发阈值
