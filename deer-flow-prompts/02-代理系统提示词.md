## 🔍 设计思路解读重点

### 设计理念1：中间件链模式

**为什么用中间件？**
- 横切关注点分离（日志、监控、安全）
- 可组合的请求处理管道
- 易于扩展新功能

**设计取舍**：
- 优点：解耦、可复用、易测试
- 缺点：执行顺序敏感、调试困难
- 替代方案：装饰器模式、AOP

### 设计理念2：自定义 Reducer

**为什么需要自定义 reducer？**
- LangGraph 默认的 dict merge 不够智能
- 需要去重、排序、冲突解决
- artifacts 需要深度合并

**设计难点**：
- 消息去重：依赖 message_id
- 顺序保证：append-in-order
- 幂等性：多次应用结果相同

### 设计理念3：LLM 驱动的记忆提取

**为什么用 LLM 提取记忆？**
- 非结构化对话 → 结构化事实
- 理解上下文语义
- 提取隐性知识

**设计挑战**：
- 幻觉问题：LLM 可能提取不存在的事实
- 成本问题：每次调用消耗 token
- 延迟问题：异步处理避免阻塞

### 设计理念4：状态机设计

**为什么用 LangGraph？**
- 可视化的状态转换
- 条件路由支持
- 检查点集成
- 时间旅行调试

**状态设计**：
- `agent` 节点：LLM 决策
- `tools` 节点：执行工具
- 条件边：`should_continue` 判断是否继续
