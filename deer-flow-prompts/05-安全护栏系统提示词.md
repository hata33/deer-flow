```python
# 寻找以下模式：
class AllowlistProvider(GuardrailProvider):
    def __init__(self, allowed_tools: list[str]):
        self.allowed_tools = allowed_tools

    async def authorize_tool_call(self, tool_name, tool_args, context):
        if tool_name in self.allowed_tools:
            return AuthorizationResult(allowed=True)
        return AuthorizationResult(
            allowed=False,
            reason=f"Tool '{tool_name}' is not in the allowlist"
        )
```

### 3. GuardrailMiddleware 集成（middleware.py）

**分析重点**：
```python
# 寻找以下模式：
class GuardrailMiddleware(Middleware):
    def __init__(self, providers: list[GuardrailProvider]):
        self.providers = providers

    async def before_tool_call(self, context: MiddlewareContext):
        tool_call = context.tool_call
        for provider in self.providers:
            result = await provider.authorize_tool_call(
                tool_call.name,
                tool_call.arguments,
                context
            )
            if not result.allowed:
                raise GuardrailException(result.reason)
```

## 🔍 设计思路解读重点

### 设计理念1：Provider 模式

**为什么用 Provider 模式？**
- 可扩展：支持多种护栏策略
- 可组合：多个 Provider 组合使用
- 可测试：每个 Provider 可独立测试

**设计权衡**：
- 优点：开闭原则、策略模式
- 缺点：增加抽象层
- 替代方案：装饰器、AOP

### 设计理念2：前置拦截

**为什么在工具调用前拦截？**
- 防止危险操作执行
- 节省资源（不执行无权限的操作）
- 明确的权限边界

**设计挑战**：
- 如何定义"危险操作"？
- 如何平衡安全性与可用性？
- 如何处理动态权限？

### 设计理念3：白名单机制

**为什么用白名单而非黑名单？**
- 默认拒绝更安全
- 明确声明允许的操作
- 避免遗漏危险操作

**设计取舍**：
- 优点：安全性高
- 缺点：配置繁琐
- 混合模式：白名单 + 黑名单

### 设计理念4：中间件集成

**为什么用中间件集成护栏？**
- 与其他横切关注点统一
- 易于插入执行流程
- 可配置执行顺序

**设计权衡**：
- 优点：一致性、可组合
- 缺点：中间件顺序敏感
- 替代方案：Agent 内部检查、工具层检查
