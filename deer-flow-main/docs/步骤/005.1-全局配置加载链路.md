承接 005.5 中 `get_app_config()` 的调用，本步骤展开全局配置从文件读取到解析完成的完整链路

---

**入口：`get_app_config()`**

``` python
def get_app_config() -> AppConfig:
    if _app_config is not None and _app_config_is_custom:
        return _app_config

    resolved_path = AppConfig.resolve_config_path()
    current_mtime = _get_config_mtime(resolved_path)

    should_reload = _app_config is None or _app_config_path != resolved_path or _app_config_mtime != current_mtime
    if should_reload:
        _load_and_cache_app_config(str(resolved_path))
    return _app_config
```

单例 + 热重载模式：首次调用时加载，后续调用直接返回缓存。当文件路径变化或 mtime 增大时自动重新加载，不需要重启服务

缓存变量：
- `_app_config` — 缓存的 AppConfig 实例
- `_app_config_path` — 上次加载时的文件路径
- `_app_config_mtime` — 上次加载时的文件修改时间
- `_app_config_is_custom` — 是否为手动注入的自定义实例（跳过热重载）

---

**第一步：解析配置文件路径**

``` python
@classmethod
def resolve_config_path(cls, config_path: str | None = None) -> Path:
```

按优先级查找 `config.yaml`：

1. 显式传入的 `config_path` 参数
2. `DEER_FLOW_CONFIG_PATH` 环境变量
3. 当前目录下的 `config.yaml`
4. 父目录下的 `config.yaml`（推荐位置，项目根目录）

找不到则抛 `FileNotFoundError`

---

**第二步：读取文件并解析 YAML**

``` python
resolved_path = cls.resolve_config_path(config_path)
with open(resolved_path, encoding="utf-8") as f:
    config_data = yaml.safe_load(f) or {}
```

标准的 YAML 读取，`yaml.safe_load` 将 YAML 转为 Python dict

---

**第三步：检查配置版本**

``` python
cls._check_config_version(config_data, resolved_path)
```

对比用户 `config.yaml` 中的 `config_version` 和 `config.example.yaml` 中的版本号。用户版本落后时打印警告，提示运行 `make config-upgrade` 合并新字段。缺少 `config_version` 视为版本 0

---

**第四步：解析环境变量**

``` python
config_data = cls.resolve_env_variables(config_data)
```

递归遍历整个 config dict，将 `$` 开头的字符串替换为对应的环境变量值：

``` python
if isinstance(config, str):
    if config.startswith("$"):
        env_value = os.getenv(config[1:])
        if env_value is None:
            raise ValueError(f"Environment variable {config[1:]} not found")
        return env_value
```

例如 `api_key: $DEEPSEEK_API_KEY` → 读取 `os.getenv("DEEPSEEK_API_KEY")` 的值替换。环境变量不存在时直接抛异常

对 dict 和 list 递归处理，确保嵌套配置中的 `$` 变量也被解析

---

**第五步：分发子配置模块**

``` python
if "title" in config_data:
    load_title_config_from_dict(config_data["title"])
if "summarization" in config_data:
    load_summarization_config_from_dict(config_data["summarization"])
if "memory" in config_data:
    load_memory_config_from_dict(config_data["memory"])
if "subagents" in config_data:
    load_subagents_config_from_dict(config_data["subagents"])
if "tool_search" in config_data:
    load_tool_search_config_from_dict(config_data["tool_search"])
if "guardrails" in config_data:
    load_guardrails_config_from_dict(config_data["guardrails"])
...
```

各子模块有独立的配置类和加载函数，这里按 key 分发到各自的 `load_*_from_dict()` 中缓存。这些子模块（title、summarization、memory 等）在项目其他地方直接通过 `get_title_config()`、`get_summarization_config()` 等函数获取，不再每次重读 config.yaml

---

**第六步：加载扩展配置（单独文件）**

``` python
extensions_config = ExtensionsConfig.from_file()
config_data["extensions"] = extensions_config.model_dump()
```

`ExtensionsConfig`（MCP servers 和 skills 启用状态）存储在独立的 `extensions_config.json` 中，不混在 `config.yaml` 里。这里单独读取后注入到 config_data 中

---

**第七步：Pydantic 校验生成 AppConfig**

``` python
result = cls.model_validate(config_data)
return result
```

将解析后的 dict 交给 Pydantic 的 `model_validate` 校验，生成 `AppConfig` 实例。`AppConfig` 的主要字段：

- `models: list[ModelConfig]` — 模型列表，每个模型有 `name`、`use`（类路径）、`model`（模型名）、`supports_thinking`、`supports_vision`、`when_thinking_enabled` 等
- `tools: list[ToolConfig]` — 工具列表
- `tool_groups: list[ToolGroupConfig]` — 工具分组
- `sandbox: SandboxConfig` — 沙箱配置
- `skills: SkillsConfig` — 技能路径配置
- `extensions: ExtensionsConfig` — MCP + skills 启用状态
- `token_usage: TokenUsageConfig` — token 用量追踪
- `tool_search: ToolSearchConfig` — 工具搜索配置
- `model_config = ConfigDict(extra="allow", frozen=False)` — 允许额外字段、允许修改

`extra="allow"` 让未在 Pydantic 模型中定义的配置项（如 provider 特定参数 `timeout`、`max_retries`）也能保留下来，后续传给模型构造函数

---

**第八步：获取具体模型配置**

``` python
def get_model_config(self, name: str) -> ModelConfig | None:
    return next((model for model in self.models if model.name == name), None)
```

按 `name` 从 `models` 列表中查找。在 007 中 `_resolve_model_name()` 和 `create_chat_model()` 都会调用此方法获取模型的完整配置（provider 类路径、参数、能力标记等）

---

**完整链路总结**：

```
get_app_config()
  → 检查缓存（单例 + mtime 热重载）
  → resolve_config_path()        查找 config.yaml 文件
  → yaml.safe_load()             读取为 Python dict
  → _check_config_version()      对比 example 版本号
  → resolve_env_variables()      递归替换 $ 环境变量
  → load_*_from_dict()           分发子配置模块各自缓存
  → ExtensionsConfig.from_file() 加载 extensions_config.json
  → model_validate()             Pydantic 校验生成 AppConfig
  → 缓存 + 返回
```

> 本步骤：`get_app_config()` 从 `config.yaml` 读取 YAML → 替换环境变量 → 分发子模块 → Pydantic 校验 → 缓存单例。之后通过 `app_config.get_model_config(name)` 按名称获取具体模型配置，供 `create_chat_model()` 反射创建 LLM 实例
