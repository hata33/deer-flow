# 反射与配置驱动设计

DeerFlow 通过字符串路径 + 运行时反射实现配置驱动：用户改 YAML 就能切换模型、工具、沙箱，不改 Python 代码。

---

## 一、为什么用字符串路径

**核心原因：config.yaml 是纯文本，不能写 Python import 语句。**

```yaml
# 用户想换模型？改一行就行
models:
  - name: gpt-4o
    use: "langchain_openai:ChatOpenAI"       # OpenAI
  - name: claude
    use: "langchain_anthropic:ChatAnthropic"  # 换成 Anthropic
```

如果代码里直接 `from langchain_openai import ChatOpenAI`，用户换模型必须改 Python 源码。用字符串路径把"用什么"从代码中抽离到配置文件，实现：

| 好处 | 说明 |
|------|------|
| 配置驱动 | 改 YAML 切换实现，不动代码 |
| 延迟加载 | 只在用到时才 import，不会因没装某个 provider 就启动失败 |
| 解耦 | 代码只依赖基类（BaseChatModel 等），不依赖具体实现 |
| 类型安全 | 反射时有 issubclass 校验，配错了启动即报错 |
| 可扩展 | 第三方写新 provider，用户加一行 YAML 就能用 |

## 二、反射模块（reflection/resolvers.py）

两个核心函数：

### resolve_variable — 解析变量

```python
resolve_variable("langchain_openai:ChatOpenAI")
# → "langchain_openai".rsplit(":") → ("langchain_openai", "ChatOpenAI")
# → import_module("langchain_openai")
# → getattr(module, "ChatOpenAI")
# → 返回 ChatOpenAI 类对象
```

解析过程：

```
"package.module:variable_name"
       │
       ├─ rsplit(":") → module_path, variable_name
       ├─ import_module(module_path) → module 对象
       ├─ getattr(module, variable_name) → 变量/类
       └─ isinstance(variable, expected_type) → 类型校验
```

可选参数 `expected_type` 做 isinstance 校验，不匹配时抛 ValueError。

### resolve_class — 解析类

```python
resolve_class("langchain_openai:ChatOpenAI", BaseChatModel)
# → resolve_variable("langchain_openai:ChatOpenAI", expected_type=type)
# → issubclass(ChatOpenAI, BaseChatModel) 校验
# → 返回 ChatOpenAI 类
```

比 `resolve_variable` 多一步 issubclass 校验，确保是类且是目标基类的子类。

### 缺失依赖提示

导入失败时不是报一个看不懂的 ImportError，而是生成可操作的安装提示：

```
Could not import module langchain_google_genai.
Missing dependency 'langchain_google_genai'.
Install it with `uv add langchain-google-genai` (or `pip install langchain-google-genai`), then restart DeerFlow.
```

通过 `MODULE_TO_PACKAGE_HINTS` 映射表把 Python 模块名（下划线）转成包名（连字符）。

## 三、配置中的 use 路径

config.yaml 中所有 `use` 字段都走反射解析：

### 模型

```yaml
models:
  - name: gpt-4o
    use: "langchain_openai:ChatOpenAI"                # 第三方 provider
  - name: claude
    use: "langchain_anthropic:ChatAnthropic"
  - name: deepseek
    use: "deerflow.models.patched_deepseek:PatchedChatDeepSeek"  # 项目内 patch
```

调用点：`models/factory.py` → `resolve_class(model_config.use, BaseChatModel)`

### 工具

```yaml
tools:
  - use: "deerflow.community.ddg_search.tools:web_search_tool"
  - use: "deerflow.community.jina_ai.tools:web_fetch_tool"
  - use: "deerflow.sandbox.tools:bash_tool"
```

调用点：`tools/tools.py` → `resolve_variable(tool.use, BaseTool)`

### 沙箱

```yaml
sandbox:
  use: "deerflow.sandbox.local:LocalSandboxProvider"
  # use: "deerflow.community.aio_sandbox:AioSandboxProvider"  # Docker 模式
```

调用点：`sandbox/sandbox_provider.py` → `resolve_class(config.sandbox.use, SandboxProvider)`

### 护栏

```yaml
guardrails:
  provider:
    use: "deerflow.guardrails.builtin:AllowlistProvider"
    # use: "aport_guardrails.providers.generic:OAPGuardrailProvider"
```

调用点：`guardrails/middleware.py` → `resolve_variable(provider.use)`

## 四、模型工厂（models/factory.py）

`create_chat_model()` 是反射最复杂的使用场景，展示完整的"字符串 → 对象"链路：

```
config.yaml                     factory.py
──────────                      ──────────
name: gpt-4o                    name = "gpt-4o"
use: "langchain_openai:ChatOpenAI"
max_tokens: 4096
                                config = get_app_config()
                                model_config = config.get_model_config("gpt-4o")
                                    │
                                    ├─ use: "langchain_openai:ChatOpenAI"
                                    ├─ max_tokens: 4096
                                    ├─ supports_thinking: false
                                    └─ ...

                                model_class = resolve_class(
                                    "langchain_openai:ChatOpenAI",
                                    BaseChatModel
                                )
                                → ChatOpenAI 类

                                model_settings = model_config.model_dump(
                                    exclude={"use", "name", "supports_thinking", ...}
                                )
                                → {"max_tokens": 4096, "model": "gpt-4o", ...}

                                model_instance = ChatOpenAI(**model_settings)
                                → 返回 BaseChatModel 实例
```

### thinking 模式的参数合并

```yaml
# config.yaml
models:
  - name: claude
    use: "langchain_anthropic:ChatAnthropic"
    supports_thinking: true
    when_thinking_enabled:
      thinking:
        type: enabled
        budget_tokens: 10000
```

```python
# thinking_enabled=True 时
model_settings.update({"thinking": {"type": "enabled", "budget_tokens": 10000}})

# thinking_enabled=False 时（显式禁用，防止模型默认启用）
kwargs.update({"thinking": {"type": "disabled"}})
```

### Codex 模型特殊处理

```python
if issubclass(model_class, CodexChatModel):
    model_settings.pop("max_tokens", None)          # Codex 不接受
    model_settings["reasoning_effort"] = "medium"    # 映射 thinking → reasoning_effort
```

## 五、设计总结

```
config.yaml（纯文本）
  │
  │  use: "package.module:ClassName"
  │
  ▼
reflection/resolvers.py
  │
  ├─ resolve_variable(path) → 变量/函数对象
  ├─ resolve_class(path, base) → 类对象（含 issubclass 校验）
  └─ 缺失依赖 → 可操作安装提示
  │
  ▼
调用方
  ├─ models/factory.py → resolve_class(use, BaseChatModel) → 创建 LLM 实例
  ├─ tools/tools.py → resolve_variable(use, BaseTool) → 加载工具
  ├─ sandbox/sandbox_provider.py → resolve_class(use, SandboxProvider) → 创建沙箱
  └─ guardrails/middleware.py → resolve_variable(use) → 加载护栏
```

本质是**依赖注入**：字符串是 key，反射是 resolver，运行时解析成实际对象。配置文件是唯一的"真相来源"，代码只依赖抽象基类，不依赖具体实现。

---

> 本文档：DeerFlow 的反射系统通过 `resolve_variable` / `resolve_class` 将 config.yaml 中的字符串路径（如 `"langchain_openai:ChatOpenAI"`）在运行时解析为 Python 对象。这使得模型、工具、沙箱、护栏的切换完全配置驱动——改 YAML 不改代码。反射包含类型校验和缺失依赖提示，模型工厂还额外处理 thinking 模式、reasoning_effort、Codex 特殊映射等能力差异。
