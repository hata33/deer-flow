# 工具调用链路：从 LLM 决定调用到 ToolMessage 回传

LLM 产出 `tool_calls` 后，经过中间件拦截、实际执行、结果回传、推流通知前端的完整链路。

---

## 一、整体链路

```
LLM 输出 AIMessage(tool_calls=[{name, args, id}])
        │
        ▼
after_model 钩子（中间件审查 tool_calls）
  ├─ SubagentLimitMiddleware  → 截断超额 task 调用
  ├─ LoopDetectionMiddleware  → 检测循环，强停则剥离 tool_calls
        │
        ▼ （tool_calls 仍存在）
wrap_tool_call 钩子（逐个执行，每个 tool_call 独立经过）
  ├─ GuardrailMiddleware      → 授权检查，拒绝则返回 error ToolMessage
  ├─ SandboxAuditMiddleware   → bash 命令审计日志
  ├─ ToolErrorHandlingMiddleware → try/except 包裹，异常转 error ToolMessage
  ├─ ClarificationMiddleware  → ask_clarification 拦截，goto END 中断
        │
        ▼ （调用实际工具函数）
handler(request) → ToolMessage(content=结果, tool_call_id=id)
        │
        ▼
ToolMessage 追加到 state.messages → astream yield chunk
        │
        ▼
serialize(chunk) → bridge.publish() → SSE → 前端
```

---

## 二、ReAct 循环中的位置

工具调用发生在 LangGraph `agent.astream()` 内部的 ReAct 循环中：

```
before_agent 钩子（ThreadData → Uploads → Sandbox）
        │
        ▼
wrap_model_call 钩子（DanglingToolCall → Summarization → ViewImage）
        │
        ▼
LLM 调用 → AIMessage
        │
        ├─ 无 tool_calls → after_agent（Title → Memory）→ 结束
        │
        └─ 有 tool_calls ──→ ★ 工具调用链路（本节）──→ 回到 wrap_model_call（第二轮 LLM）
```

每轮 LLM 输出 `tool_calls`，执行完所有工具后，ToolMessage 追加到 messages，进入下一轮 LLM 调用。循环直到 LLM 不再输出 `tool_calls`。

---

## 三、after_model 阶段：审查 tool_calls

LLM 输出后、工具执行前，中间件可以审查和修改 `tool_calls`。

### 3.1 SubagentLimitMiddleware

**钩子**：`after_model`
**作用**：截断超额的 `task` 工具调用

```
LLM 输出 5 个 task 调用 → 保留前 3 个（MAX_CONCURRENT_SUBAGENTS）→ 丢弃后 2 个
```

实现：统计 AIMessage 中 `name == "task"` 的 tool_calls，超出 `max_concurrent`（默认 3，钳制到 [2,4]）的用 `model_copy(update={"tool_calls": truncated})` 替换原消息。

### 3.2 LoopDetectionMiddleware

**钩子**：`after_model`
**作用**：检测重复工具调用循环

```
滑动窗口（20 次调用）→ 对 tool_calls 做 MD5 哈希
  ├─ 相同哈希 ≥ 3 次（warn_threshold）→ 注入 HumanMessage 警告
  └─ 相同哈希 ≥ 5 次（hard_limit）→ 剥离所有 tool_calls，强制 LLM 输出文本
```

按 `thread_id` 独立跟踪，LRU 驱动清理。这是 P0 安全保障，防止 agent 无限循环直到递归限制。

---

## 四、wrap_tool_call 阶段：逐个执行

每个 `tool_call` 独立经过 `wrap_tool_call` 钩子链。LangGraph 的 `ToolNode` 逐个调用：

```
tool_calls = [{name: "search", id: "call_1"}, {name: "bash", id: "call_2"}]

call_1 → Guardrail → Audit → ErrorHandle → Clarify → handler() → ToolMessage
call_2 → Guardrail → Audit → ErrorHandle → Clarify → handler() → ToolMessage
```

### 4.1 GuardrailMiddleware

**钩子**：`wrap_tool_call`
**作用**：工具调用授权

```
检查 tool_call 是否在允许列表中
  ├─ 允许 → handler(request) 正常执行
  └─ 拒绝 → 返回 error ToolMessage（不执行工具）
```

可选中间件，通过 `config.yaml` 的 `guardrails.enabled` 开启。

### 4.2 SandboxAuditMiddleware

**钩子**：`wrap_tool_call`
**作用**：记录 bash 工具的命令审计日志

仅对 `name == "bash"` 的工具调用记录命令和参数，不影响执行流程。

### 4.3 ToolErrorHandlingMiddleware

**钩子**：`wrap_tool_call`
**作用**：异常兜底，将工具异常转为 error ToolMessage

```python
try:
    return handler(request)        # 正常执行工具
except GraphBubbleUp:
    raise                          # 保留 LangGraph 控制流信号（中断/暂停）
except Exception as exc:
    return ToolMessage(
        content=f"Error: Tool '{name}' failed with {type}: {detail}",
        tool_call_id=tc_id,
        name=tool_name,
        status="error",
    )
```

关键：`GraphBubbleUp` 不捕获（LangGraph 的中断/恢复信号），其他所有异常转为 error ToolMessage，LLM 看到"工具失败了"后可以选择替代方案继续。

### 4.4 ClarificationMiddleware

**钩子**：`wrap_tool_call`
**作用**：拦截 `ask_clarification`，中断执行等待用户回复

```python
if request.tool_call.get("name") != "ask_clarification":
    return handler(request)        # 其他工具正常执行

# ask_clarification → 格式化问题 → Command(goto=END) 中断
return Command(
    update={"messages": [ToolMessage(content=格式化问题, tool_call_id=id)]},
    goto=END,
)
```

这是中间件链的最后一道关卡。中断后前端展示澄清问题，用户回复后恢复执行。

---

## 五、工具实际执行

经过所有中间件后，`handler(request)` 调用实际工具函数：

```python
# handler 由 LangGraph 的 ToolNode 提供
# request.tool_call = {name: "search", args: {query: "X"}, id: "call_1"}

tool_result = tool_function(**request.tool_call["args"])
→ ToolMessage(content=tool_result, tool_call_id="call_1", name="search")
```

工具函数来自 `get_available_tools()` 组装的 6 层来源：

| 来源 | 示例 | 特点 |
|------|------|------|
| config.yaml 定义 | tavily_search | `resolve_variable` 动态加载 |
| MCP 工具 | 外部服务工具 | 延迟加载，mtime 缓存失效 |
| 内置工具 | present_files, ask_clarification | 始终可用 |
| 条件工具 | view_image（vision 模型）, task（subagent_enabled） | 按条件注入 |
| 社区工具 | jina_ai, firecrawl | 可选安装 |
| ACP 工具 | 外部 agent 调用 | 通过 ACP 协议 |

## 六、ToolMessage 回传与推流

工具执行完成后，ToolMessage 追加到 `state.messages`，LangGraph `astream` yield 这个状态变化作为 chunk：

### values 模式

```json
// 完整状态快照，messages 包含新增的 ToolMessage
{"messages": [HumanMessage, AIMessage(tool_calls=[...]), ToolMessage(content="结果")]}
```

### messages 模式

```json
// 增量消息元组
[ToolMessage(content="结果", name="search", tool_call_id="call_1"), {"langgraph_node": "tools"}]
```

### 推流路径

```
ToolMessage 追加到 state
  → agent.astream() yield chunk
  → serialize(chunk, mode) → JSON
  → bridge.publish(run_id, sse_event, data)
  → asyncio.Queue
  → sse_consumer: queue.get() → format_sse()
  → SSE 帧推前端
```

前端 `useStream` hook 按 `event` 名分发：
- `messages` 事件 → 检查 `type === "tool"` → 展示工具结果
- `values` 事件 → 更新完整状态

## 七、完整时序：一次工具调用

```
LLM 调用
  │
  ▼
AIMessage(tool_calls=[{name:"search", args:{query:"X"}, id:"c1"}])
  │
  ├─ after_model:
  │   SubagentLimitMiddleware → 非 task 工具，不截断
  │   LoopDetectionMiddleware → 首次调用，无循环
  │
  ├─ wrap_tool_call (call c1):
  │   GuardrailMiddleware → search 在允许列表 → pass
  │   SandboxAuditMiddleware → 非 bash 工具 → pass
  │   ToolErrorHandlingMiddleware → try 包裹
  │   ClarificationMiddleware → 非 ask_clarification → pass
  │   │
  │   handler(request) → search(query="X") → "搜索结果..."
  │   │
  │   → ToolMessage(content="搜索结果...", tool_call_id="c1", name="search")
  │
  ▼
ToolMessage 追加到 state.messages
  │
  ▼
agent.astream() yield chunk
  │
  ├─ serialize(chunk, mode="messages")
  │   → [ToolMessage序列化, {langgraph_node: "tools"}]
  │
  ├─ bridge.publish(run_id, "messages", data)
  │   → queue.put(StreamEvent)
  │
  ▼
sse_consumer → format_sse("messages", data)
  → 前端收到工具结果
  │
  ▼
回到 wrap_model_call → 第二轮 LLM（带工具结果的上下文）
```

## 八、异常场景

| 场景 | 触发点 | 行为 | LLM 收到什么 |
|------|--------|------|-------------|
| 工具执行异常 | handler() 抛异常 | ToolErrorHandling 捕获 | error ToolMessage："Tool 'search' failed..." |
| 工具被护栏拒绝 | GuardrailMiddleware | 不执行，返回 error ToolMessage | error ToolMessage："Tool not authorized" |
| 用户中断（dangling） | 下次请求 | DanglingToolCallMiddleware 补丁 | 占位 ToolMessage："Tool call was interrupted..." |
| 循环检测警告 | after_model | 注入 HumanMessage 警告 | "You are repeating..." |
| 循环检测强停 | after_model | 剥离所有 tool_calls | 无 tool_calls，LLM 必须输出文本 |
| 澄清请求 | ClarificationMiddleware | Command(goto=END) 中断 | 前端展示问题，等待用户回复 |

所有场景都保证 LLM 收到格式正确的消息序列——每个 tool_call 都有对应的 ToolMessage，不会因缺失响应而报错。

---

## 九、中间件钩子与工具调用的对应关系

| 钩子 | 时机 | 作用于 tool_calls 的中间件 |
|------|------|--------------------------|
| `wrap_model_call` | LLM 调用前 | DanglingToolCallMiddleware（修补悬空调用） |
| `after_model` | LLM 输出后 | SubagentLimitMiddleware（截断超额 task）、LoopDetectionMiddleware（检测循环） |
| `wrap_tool_call` | 每个工具执行时 | GuardrailMiddleware（授权）、SandboxAuditMiddleware（审计）、ToolErrorHandlingMiddleware（异常兜底）、ClarificationMiddleware（拦截澄清） |
| `after_agent` | Agent 循环结束 | 无直接作用于工具调用 |

## 十、前端消息分组（groupMessages）

前端收到 SSE 事件后，`useStream` hook 更新 `thread.messages`。`MessageList` 组件调用 `groupMessages()` 将消息列表分为 6 种分组：

| 分组类型 | 触发条件 | 包含的消息 |
|---------|---------|----------|
| `human` | `message.type === "human"` | 用户消息 |
| `assistant` | AIMessage 有 content 且无 tool_calls | 最终回复气泡 |
| `assistant:processing` | AIMessage 有 `reasoning` 或 `tool_calls`（非特殊工具） | AI 思考 + 工具调用 + 工具结果 |
| `assistant:present-files` | AIMessage 的 tool_calls 含 `present_files` | 文件展示 |
| `assistant:clarification` | ToolMessage 的 `name === "ask_clarification"` | 澄清问题 |
| `assistant:subagent` | AIMessage 的 tool_calls 含 `task` | 子智能体任务卡片 |

### 分组逻辑核心流程

```
遍历 messages：
  human → 新建 human 组
  tool:
    ├─ ask_clarification → 归入当前 processing 组 + 新建 clarification 组
    └─ 其他工具 → 归入当前 processing 组（lastOpenGroup）
  ai:
    ├─ 有 present_files → 新建 present-files 组
    ├─ 有 task → 新建 subagent 组
    ├─ 有 reasoning 或 tool_calls → 新建/追加到 processing 组
    └─ 有 content 且无 tool_calls → 新建 assistant 组（最终回复）
```

关键：ToolMessage 不会独立成组，总是归入前一个打开的 processing 组。这保证了工具调用（AIMessage 的 tool_calls）和工具结果（ToolMessage）在同一个组内。

---

## 十一、工具调用的渲染（MessageGroup 组件）

`assistant:processing` 分组由 `MessageGroup` 组件渲染。它将分组内的消息转换为 `ChainOfThought` 步骤列表：

### convertToSteps 转换

```typescript
for (const message of messages) {
  if (message.type === "ai") {
    // 1. 提取思考过程
    reasoning → CoTReasoningStep { type: "reasoning", reasoning: "..." }

    // 2. 提取每个 tool_call（task 工具跳过，由 SubtaskCard 单独处理）
    for (const tool_call of message.tool_calls) {
      → CoTToolCallStep {
          type: "toolCall",
          name: "search",
          args: { query: "X" },
          result: findToolCallResult(tool_call_id, messages)  // 从同组 ToolMessage 找结果
        }
    }
  }
}
```

### 工具类型 → 渲染映射

`ToolCall` 组件根据 `name` 选择不同的 UI：

| 工具名 | 图标 | 渲染 |
|--------|------|------|
| `web_search` | SearchIcon | "在网络上搜索 {query}" + 搜索结果链接列表 |
| `image_search` | SearchIcon | "搜索相关图片" + 缩略图网格 |
| `web_fetch` | GlobeIcon | "查看网页" + 可点击链接 |
| `bash` | SquareTerminalIcon | description 标签 + bash 代码块 |
| `read_file` | BookOpenTextIcon | description 标签 + 文件路径 |
| `write_file` / `str_replace` | NotebookPenIcon | description 标签 + 点击打开 Artifact 编辑器 |
| `ls` | FolderOpenIcon | description 标签 + 目录路径 |
| `ask_clarification` | QuestionIcon | "需要你的帮助" |
| `write_todos` | ListTodoIcon | "写入待办" |
| `task` | — | 跳过（由 SubtaskCard 组件处理） |
| 其他 | WrenchIcon | "使用工具 {name}" |

### 折叠与展开

工具步骤默认展示最后一个（`lastToolCallStep`），更早的步骤通过 "展开 N 个步骤" 按钮显示。思考过程（reasoning）也默认折叠，点击展开。

## 十二、子智能体任务的渲染（SubtaskCard）

`assistant:subagent` 分组由 `MessageList` 直接处理，不经过 `MessageGroup`：

```
AIMessage(tool_calls=[{name:"task", args:{description, prompt, subagent_type}}])
  → 创建 Subtask { id, subagent_type, description, status: "in_progress" }
  → 渲染 SubtaskCard

ToolMessage(content="Task Succeeded. Result: ...")
  → updateSubtask { id, status: "completed", result: "..." }
  → SubtaskCard 更新状态
```

任务状态从 ToolMessage 的 content 前缀推断：
- `"Task Succeeded. Result:"` → completed
- `"Task failed."` → failed
- `"Task timed out"` → failed
- 其他 → in_progress

## 十三、端到端示例：一次搜索工具调用

```
后端：
  LLM → AIMessage(tool_calls=[{name:"web_search", args:{query:"量子计算"}, id:"c1"}])
  → after_model 审查通过
  → wrap_tool_call: handler() → 搜索结果 JSON
  → ToolMessage(content='[{title:"...", url:"..."}]', tool_call_id="c1")
  → serialize → bridge.publish → SSE

前端：
  useStream 收到 messages 事件 → messages 更新

  groupMessages():
    AIMessage(tool_calls=[...]) → assistant:processing 组
    ToolMessage(content=搜索结果) → 归入同一个 processing 组

  MessageGroup 组件:
    convertToSteps():
      AIMessage → CoTToolCallStep { name: "web_search", args: {query:"量子计算"}, result: [...] }
    ToolCall 组件:
      name === "web_search"
      → 渲染 "在网络上搜索 量子计算" + 搜索结果链接列表
```

---

> 本文档：工具调用从 LLM 输出 `tool_calls` 开始，经过 after_model 审查（限额、循环检测）→ wrap_tool_call 执行（授权、审计、异常兜底、澄清拦截）→ ToolMessage 回传 → serialize → bridge.publish → SSE 推前端。前端通过 `groupMessages` 将消息分为 6 种分组，`MessageGroup` 组件将工具调用转为 `ChainOfThought` 步骤列表，根据工具名选择不同图标和渲染方式。子智能体任务由 `SubtaskCard` 独立处理状态。
