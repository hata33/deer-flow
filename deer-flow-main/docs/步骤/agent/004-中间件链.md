中间件链——14 个中间件的有序链与两种构建方式

---

## 一、中间件顺序与职责

**文件**：`agents/lead_agent/agent.py` → `_build_middlewares()`

中间件按固定顺序组装，顺序有严格依赖关系（如 ThreadData 必须在 Sandbox 之前，Clarification 必须在最后）：

```
 0. ThreadDataMiddleware          → before_agent: 初始化 thread 工作目录路径
 1. UploadsMiddleware             → before_agent: 处理上传文件元数据注入到消息
 2. SandboxMiddleware             → before_agent: 沙箱环境创建与配置
 3. DanglingToolCallMiddleware    → wrap_tool_call: 修补缺失的 ToolMessage（配对悬挂的 tool_call）
 4. ToolErrorHandlingMiddleware   → wrap_tool_call: 工具异常转 ToolMessage（不让异常中断流程）
 5. SummarizationMiddleware       → before_model: 上下文超长时自动摘要压缩（可选）
 6. TodoMiddleware                → before_model + 工具注入: 计划模式任务管理（可选）
 7. TokenUsageMiddleware          → after_model: token 用量统计（可选）
 8. TitleMiddleware               → after_model: 首轮对话后自动生成标题
 9. MemoryMiddleware              → after_agent: 对话完成后排队更新记忆
10. ViewImageMiddleware           → before_model: 注入图片 base64 到消息（可选）
11. DeferredToolFilterMiddleware  → before_model: 延迟工具 schema 从模型绑定中隐藏（可选）
12. SubagentLimitMiddleware       → wrap_tool_call: 截断超出限制的并行 task 调用（可选）
13. LoopDetectionMiddleware       → after_model: 检测并打断重复工具调用循环
14. ClarificationMiddleware       → wrap_tool_call: 拦截 ask_clarification 并中断流程（始终最后）
```

### 每个中间件的钩子方法

LangGraph 中间件有四个钩子点：

| 钩子 | 时机 | 典型用途 |
|------|------|----------|
| `before_agent()` | agent 执行前 | 初始化环境、注入状态 |
| `before_model()` | LLM 调用前 | 修改输入、压缩上下文 |
| `after_model()` | LLM 响应后 | 生成标题、检测循环、统计用量 |
| `wrap_tool_call()` | 工具执行时 | 拦截、异常处理、截断 |
| `after_agent()` | agent 执行后 | 更新记忆 |

## 二、关键中间件详解

### 2.1 ThreadDataMiddleware（#0）

**文件**：`middlewares/thread_data_middleware.py`

在 `before_agent()` 中计算或创建 thread 的工作目录：

```
{base_dir}/threads/{thread_id}/user-data/workspace    ← 临时工作
{base_dir}/threads/{thread_id}/user-data/uploads      ← 用户上传
{base_dir}/threads/{thread_id}/user-data/outputs      ← 最终输出
```

将路径写入 `state.thread_data`。`lazy_init=True`（默认）时只计算路径不创建目录

### 2.2 TitleMiddleware（#8）

**文件**：`middlewares/title_middleware.py`

在 `after_model()` 中触发。条件：`state.title` 为空 + 有且仅有 1 条 human + 至少 1 条 ai

```
_should_generate_title(state)
  ├─ config.enabled → False 则跳过
  ├─ state.title 已存在 → 跳过
  └─ 只有第一轮对话（1 human + ≥1 ai）→ 触发

_build_title_prompt(state)
  ├─ 提取首条 human 和 ai 消息（截断 500 字）
  └─ 填入 config.prompt_template

_parse_title(content) → 去引号、截断到 max_chars
_fallback_title(user_msg) → LLM 失败时取前 50 字符
```

### 2.3 LoopDetectionMiddleware（#13）

**文件**：`middlewares/loop_detection_middleware.py`

检测策略：对每次 AI 响应中的 tool_calls 做 MD5 哈希，在滑动窗口内计数

```
warn_threshold=3 → 注入 "你在重复自己——收尾" 的 HumanMessage（每哈希仅一次）
hard_limit=5     → 剥离所有 tool_calls，强制输出文本答案
window_size=20   → 跟踪最近 20 次工具调用
```

为什么用 HumanMessage 而非 SystemMessage：Anthropic 模型要求系统消息只在对话开头出现，中间注入会崩溃

### 2.4 ClarificationMiddleware（#14）

**文件**：`middlewares/clarification_middleware.py`

在 `wrap_tool_call()` 中拦截 `ask_clarification` 工具调用：

```
LLM 调用 ask_clarification(question, clarification_type, context, options)
  │
  ├─ ClarificationMiddleware.wrap_tool_call()
  │   ├─ 检测 name == "ask_clarification"
  │   ├─ 格式化为用户友好消息（含类型图标 ❓🤔🔀⚠️💡）
  │   └─ 返回 Command(update={messages: [ToolMessage]}, goto=END)
  │
  └─ 执行中断，等待用户回复后继续
```

不拦截的工具调用正常传递给 `handler(request)` 执行

## 三、两种构建方式

### 3.1 Gateway 方式（make_lead_agent）

**文件**：`agents/lead_agent/agent.py` → `_build_middlewares()`

硬编码顺序，条件注入：

```python
def _build_middlewares(config, model_name, agent_name, custom_middlewares):
    middlewares = build_lead_runtime_middlewares(lazy_init=True)  # 0-4: 基础中间件

    if summarization_enabled:
        middlewares.append(SummarizationMiddleware(...))
    if is_plan_mode:
        middlewares.append(TodoMiddleware(...))
    if token_usage_enabled:
        middlewares.append(TokenUsageMiddleware())

    middlewares.append(TitleMiddleware())
    middlewares.append(MemoryMiddleware(agent_name=agent_name))

    if supports_vision:
        middlewares.append(ViewImageMiddleware())
    if tool_search_enabled:
        middlewares.append(DeferredToolFilterMiddleware())
    if subagent_enabled:
        middlewares.append(SubagentLimitMiddleware(...))

    middlewares.append(LoopDetectionMiddleware())

    if custom_middlewares:
        middlewares.extend(custom_middlewares)

    middlewares.append(ClarificationMiddleware())  # 始终最后
    return middlewares
```

### 3.2 SDK 方式（create_deerflow_agent）

**文件**：`agents/factory.py` → `_assemble_from_features()`

声明式特性标志 `RuntimeFeatures`：

```python
@dataclass
class RuntimeFeatures:
    sandbox: bool | AgentMiddleware = True
    memory: bool | AgentMiddleware = False
    summarization: Literal[False] | AgentMiddleware = False
    subagent: bool | AgentMiddleware = False
    vision: bool | AgentMiddleware = False
    auto_title: bool | AgentMiddleware = False
    guardrail: Literal[False] | AgentMiddleware = False
```

每个特性三种值：
- `True` → 使用内置默认中间件
- `False` → 禁用
- `AgentMiddleware` 实例 → 自定义替换

**@Next / @Prev 定位装饰器**：额外中间件通过装饰器声明在链中的位置

```python
@Next(TitleMiddleware)       # 放在 TitleMiddleware 之后
class MyCustomMiddleware(AgentMiddleware): ...

@Prev(ClarificationMiddleware)  # 放在 ClarificationMiddleware 之前
class AnotherMiddleware(AgentMiddleware): ...
```

插入算法：
1. 无锚点的额外中间件 → 插入到 ClarificationMiddleware 之前
2. 有锚点的 → 迭代查找锚点位置并插入
3. 不变量保证：ClarificationMiddleware 始终在最后（如果被推走则强制拉回）
