# DeerFlow Agent 系统总览总结

> 本文档是对 `agent/` 目录下 12 篇分步文档的高度概括，按 Agent 的**构建时**和**运行时**两个维度组织，帮助快速理解 DeerFlow 的核心运作原理。

---

## 一、整体架构概览

DeerFlow 的 Agent 系统分为两层：

```
┌─────────────────────────────────────────────────────┐
│                    Gateway 层                        │
│     路由(thread_runs.py) → Service(services.py)      │
│     接收 HTTP 请求，编排后台任务                       │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                   Runtime 层                         │
│     worker.py (执行编排) + stream_bridge (推流)        │
│     + store (元数据) + serialization (序列化)          │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                   Agent 层                           │
│     make_lead_agent() → create_agent()               │
│     模型 + 工具 + 中间件链 + 提示词 + ThreadState      │
└─────────────────────────────────────────────────────┘
```

## 二、构建时：`make_lead_agent()` 五步组装

每次对话请求都调用一次 `make_lead_agent(config)`，动态组装 LangGraph 编译图：

| 步骤 | 内容 | 详情文档 |
|------|------|----------|
| **1. 解析参数** | 从 `config.configurable` 提取 `thinking_enabled`、`model_name`、`plan_mode` 等 | [002](002-工厂函数与模型解析.md) |
| **2. 模型解析** | 三级优先级：请求参数 > 智能体配置 > 全局默认 | [002](002-工厂函数与模型解析.md) |
| **3. 工具加载** | 四层工具源：config.yaml 反射 + 内置/MCP/ACP | [003](003-工具加载.md) |
| **4. 中间件链** | 14 个有序中间件，`ClarificationMiddleware` 始终最后 | [004](004-中间件链.md) |
| **5. 提示词组装** | `SYSTEM_PROMPT_TEMPLATE` 动态填充记忆/技能/子代理 | [005](005-系统提示词.md) |

最终 `create_agent(model, tools, middleware, prompt, ThreadState)` 产出 `CompiledStateGraph`。

### 模型三级优先级
```
请求参数 model_name → 智能体配置 agent_config.model → 全局默认 models[0].name
```

### 工具四层来源
```
config.yaml 反射加载 → 内置工具(ask_clarification, present_file)
  → MCP Server 缓存工具（可延迟加载）→ ACP 外部代理工具
```

### 14 个中间件有序链
```
 0. ThreadDataMiddleware          → 工作目录初始化
 1. UploadsMiddleware             → 上传文件处理
 2. SandboxMiddleware             → 沙箱环境创建
 3. DanglingToolCallMiddleware    → 修补悬挂工具调用
 4. ToolErrorHandlingMiddleware   → 工具异常转消息
 5. SummarizationMiddleware       → 上下文压缩(可选)
 6. TodoMiddleware               → 计划模式任务(可选)
 7. TokenUsageMiddleware          → 用量追踪(可选)
 8. TitleMiddleware              → 首轮生成标题
 9. MemoryMiddleware             → 排队记忆更新
10. ViewImageMiddleware           → 图片注入(可选)
11. DeferredToolFilterMiddleware  → 延迟工具过滤(可选)
12. SubagentLimitMiddleware       → 子代理并发限制(可选)
13. LoopDetectionMiddleware       → 循环检测打断
14. ClarificationMiddleware       → 澄清请求拦截(始终最后)
```

## 三、运行时：ReAct 循环与中间件执行

`agent.astream(graph_input, config)` 启动后，进入 ReAct 循环，中间件在五个钩子点介入：

```
before_agent → before_model → [LLM] → after_model → wrap_tool_call → [Tool] → 循环
                                                                              │
after_agent ←──────────────────────────────────────────────────────────────────┘
```

每个节点执行后，LangGraph 自动写入检查点，worker 通过 `StreamBridge` 推流到前端。

## 四、系统提示词动态填充

`SYSTEM_PROMPT_TEMPLATE` 通过 `{placeholder}` 动态插入：

| 占位符 | 内容来源 | 特性 |
|--------|----------|------|
| `{agent_name}` | 自定义智能体名 | — |
| `{soul}` | SOUL.md 人格文件 | — |
| `{memory_context}` | memory.json | **构建时注入**，下一轮才看到更新 |
| `{skills_section}` | skills 目录 | Progressive Loading（只列技能路径） |
| `{deferred_tools_section}` | DeferredToolRegistry | 只列工具名，LLM 通过 tool_search 按需加载 |
| `{subagent_section}` | 动态生成 | 含硬性并发限制和编排策略 |

## 五、记忆系统：三层架构

```
MemoryMiddleware.after_agent()
  → MemoryUpdateQueue (debounce 30s)
    → MemoryUpdater (LLM 分析对话 → JSON 更新指令)
      → FileMemoryStorage (原子写入 memory.json)

下轮构建时:
  _get_memory_context() → 读取 memory.json → 注入 <memory> 段落
```

关键特性：
- **文件存储**：`memory.json`，含 user/history/facts 三大分组
- **防抖队列**：30s 无新消息才触发更新，避免连续对话频繁调用 LLM
- **延迟注入**：本轮更新，下一轮才能读到

## 六、Lead Agent 与 Subagent 协作

Lead Agent 是编排者，通过 `task` 工具将复杂任务分解委派给 Subagent：

| Subagent | 用途 | 最大轮次 | 禁止工具 |
|----------|------|----------|----------|
| `general-purpose` | 复杂多步任务（调研、分析） | 50 | task, ask_clarification, present_files |
| `bash` | 命令执行（git/build/test） | 30 | 同上 |

关键机制：
- **并发控制**：`SubagentLimitMiddleware` 截断超限 task 调用（限制 [2,4]，默认 3）
- **双线程池**：调度池(3) + 执行池(3)，提供超时控制
- **资源继承**：继承父 agent 的模型、沙箱、工作目录
- **隔离**：无对话历史、无 checkpointer、不能递归创建 subagent

## 七、StreamBridge：发布订阅模式

```
生产者 (worker.py)                    消费者 (sse_consumer)
  │                                       │
  ├─ bridge.publish(run_id, event, data)  │
  │  → asyncio.Queue.put()               │
  │                                       ├─ bridge.subscribe(run_id)
  │                                       │  → asyncio.Queue.get()
  │                                       │  → format_sse() → yield
  ├─ bridge.publish_end(run_id)          │
  │  → END_SENTINEL                      ├─ 收到 END_SENTINEL → 关闭流
```

每 run_id 独立一个 `asyncio.Queue`，容量 256。15s 无数据发心跳保活，队列满则丢弃事件（不阻塞 agent）。抽象接口 `StreamBridge` 通过工厂 `make_stream_bridge()` 创建 `MemoryStreamBridge` 实例，存入 `app.state`。

## 八、Runtime 注入机制

LangGraph 的 `Runtime` 对象是运行时上下文容器，在 `worker.py` 中创建并注入：

```python
runtime = Runtime(context={"thread_id": thread_id}, store=store)
config["configurable"]["__pregel_runtime"] = runtime
```

四个属性及其用途：

| 属性 | 存储内容 | 读取者 |
|------|----------|--------|
| `context` | thread_id, sandbox_id | 中间件(thread_id)、工具(sandbox_id) |
| `state` | 当前 ThreadState 引用 | 工具(读取 sandbox、thread_data) |
| `config` | 当前 RunnableConfig | 工具(读取 model_name、trace_id) |
| `store` | LangGraph Store 实例 | 工具(间接) |

## 九、ThreadState 状态设计

继承 `AgentState`（仅 `messages` 字段），扩展 7 个字段：

| 字段 | 归并策略 | 写入者 | 用途 |
|------|----------|--------|------|
| `messages` | 追加 | LLM/工具/中间件 | 驱动 ReAct 循环 |
| `sandbox` | 覆盖 | SandboxMiddleware | 沙箱 ID |
| `thread_data` | 覆盖 | ThreadDataMiddleware | 工作目录路径 |
| `title` | 覆盖 | TitleMiddleware | 自动标题 |
| `artifacts` | **去重合并** | present_files 工具 | 产出文件列表 |
| `todos` | 覆盖 | write_todos 工具 | 待办列表 |
| `uploaded_files` | 覆盖 | UploadsMiddleware | 上传文件元数据 |
| `viewed_images` | **字典合并/清空** | ViewImageMiddleware | 图片 base64 缓存 |

两种自定义 reducer：`merge_artifacts`（去重合并）、`merge_viewed_images`（合并 + 空字典清空释放内存）。

## 十、完整请求时间线

```
T+0s    HTTP → 路由层 → Service 层 → 启动后台 task
T+0.1s  worker.py: Runtime 注入 + make_lead_agent() 五步组装
T+0.2s  agent.astream() 开始 → before_agent 中间件
T+0.5s  第一次 LLM 调用
T+1s    LLM 返回(含 tool_calls) → after_model → wrap_tool_call
T+2s    工具执行完成 → 检查点写入 → 循环
T+3s    最终响应 → bridge.publish_end() → after_agent(MemoryMiddleware)
T+33s   防抖到期 → MemoryUpdater 更新 memory.json
T+5min  标题同步 + 资源清理
```
