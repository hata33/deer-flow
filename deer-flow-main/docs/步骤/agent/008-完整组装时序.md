## 一、端到端流程

```
前端 POST /api/threads/{thread_id}/runs/stream
  │
  ├─ 1. 路由层 [thread_runs.py]
  │   ├─ get_stream_bridge(request) → app.state.stream_bridge
  │   ├─ get_run_manager(request)   → app.state.run_manager
  │   └─ start_run(body, thread_id, request)
  │
  ├─ 2. Service 层 [services.py]
  │   ├─ get_checkpointer(request)  → app.state.checkpointer
  │   ├─ get_store(request)         → app.state.store
  │   ├─ run_mgr.create_or_reject() → 创建 RunRecord（含并发策略检查）
  │   ├─ _upsert_thread_in_store()  → 确保 thread 在 Store 可见
  │   ├─ resolve_agent_factory()    → make_lead_agent 引用
  │   ├─ normalize_input()          → 转换消息格式
  │   ├─ build_run_config()         → 构建 RunnableConfig
  │   ├─ normalize_stream_modes()   → 默认 ["values"]
  │   ├─ asyncio.create_task(run_agent(...))
  │   └─ asyncio.create_task(_sync_thread_title_after_run(...))
  │
  ├─ 3. 运行时层 [worker.py]
  │   ├─ Runtime 注入：thread context + store → config
  │   ├─ agent_factory(config) → make_lead_agent(config)
  │   ├─ stream_mode 映射：前端协议名 → LangGraph 模式
  │   └─ agent.astream(graph_input, config)
  │       │
  │       │  ← 以下在 LangGraph 编译图内部执行 →
  │       │
  │       ├─ 4. 中间件 before_agent()
  │       │   ├─ ThreadDataMiddleware   → 计算工作目录路径
  │       │   ├─ UploadsMiddleware      → 注入上传文件信息
  │       │   └─ SandboxMiddleware      → 创建沙箱环境
  │       │
  │       ├─ 5. ReAct 循环
  │       │   │
  │       │   ├─ [5a] 中间件 before_model()
  │       │   │   ├─ SummarizationMiddleware → 上下文过长时压缩
  │       │   │   ├─ TodoMiddleware          → 注入待办指令
  │       │   │   ├─ ViewImageMiddleware     → 注入图片 base64
  │       │   │   └─ DeferredToolFilterMiddleware → 隐藏延迟工具 schema
  │       │   │
  │       │   ├─ [5b] LLM 调用 → AI 响应（含 tool_calls 或纯文本）
  │       │   │
  │       │   ├─ [5c] 中间件 after_model()
  │       │   │   ├─ TokenUsageMiddleware    → 统计用量
  │       │   │   ├─ TitleMiddleware         → 首轮生成标题
  │       │   │   └─ LoopDetectionMiddleware → 检测循环
  │       │   │
  │       │   ├─ [5d] 检查是否需要工具调用
  │       │   │   ├─ 无 tool_calls → 跳到步骤 6
  │       │   │   └─ 有 tool_calls → 执行工具
  │       │   │
  │       │   ├─ [5e] 中间件 wrap_tool_call()
  │       │   │   ├─ DanglingToolCallMiddleware → 修补悬挂调用
  │       │   │   ├─ ToolErrorHandlingMiddleware → 异常转 ToolMessage
  │       │   │   ├─ SubagentLimitMiddleware    → 截断超限 task 调用
  │       │   │   └─ ClarificationMiddleware    → 拦截澄清请求
  │       │   │
  │       │   ├─ [5f] 执行工具 → ToolMessage 结果
  │       │   │
  │       │   └─ [5g] 检查点写入 → 回到 [5a]
  │       │
  │       ├─ 6. 中间件 after_agent()
  │       │   └─ MemoryMiddleware → 排队记忆更新
  │       │
  │       └─ 7. 推流收尾
  │           ├─ bridge.publish_end(run_id)
  │           ├─ run_mgr.set_status(run_id, RunStatus.success)
  │           └─ bridge.cleanup(run_id, delay=60)
  │
  └─ 8. 异步后置任务
      └─ _sync_thread_title_after_run()
          ├─ 等待 run task 完成
          ├─ checkpointer.aget_tuple() → 读取 title
          └─ store.aput() → 同步到 Store
```

## 二、五个核心对象的生命周期

| 对象 | 创建时机 | 生命周期 | 持久化 |
|------|----------|----------|--------|
| `CompiledStateGraph` (agent) | 每次 `make_lead_agent()` 调用 | 单次 `astream()` | 否（内存） |
| `RunRecord` | `start_run()` 中创建 | 运行期间 + 延迟 300s 清理 | 否（RunManager 内存字典） |
| 检查点 | `astream()` 每步自动写入 | 持久化（sqlite/postgres）或进程内（memory） | Checkpointer |
| Store 记录 | 创建 thread / upsert 时写入 | 持久化 | Store |
| 记忆 (memory.json) | 防抖队列处理后写入 | 持久化 | 文件系统 |

## 四、一轮对话的时间线

```
T+0s    HTTP 请求到达路由层
T+0s    Service 层创建 RunRecord、upsert Store
T+0s    启动后台 task（run_agent）+ 标题同步 task
T+0s    返回 StreamingResponse，SSE 连接建立

T+0.1s  run_agent 开始执行
T+0.1s  Runtime 注入 config
T+0.1s  make_lead_agent() 构建编译图
         ├─ create_chat_model()          → 创建 LLM 实例
         ├─ get_available_tools()         → 组装工具列表
         ├─ _build_middlewares()          → 构建中间件链
         └─ apply_prompt_template()       → 组装提示词（含记忆注入）

T+0.2s  agent.astream() 开始
T+0.2s  before_agent() 中间件执行
T+0.5s  第一次 LLM 调用
T+1s    LLM 返回（含 tool_calls）
T+1s    after_model() 中间件执行（TitleMiddleware 可能触发）
T+1s    wrap_tool_call() 中间件执行
T+2s    工具执行完成
T+2s    检查点自动写入
T+2s    第二次 LLM 调用
T+3s    LLM 返回最终文本响应
T+3s    bridge.publish_end() → SSE 流结束

T+3s    after_agent() → MemoryMiddleware 排队记忆更新
T+3s    run_mgr.set_status(success)
T+3s    前端收到最后一个 SSE 帧

T+33s   防抖计时器到期 → MemoryUpdater 开始处理
T+35s   LLM 分析对话 → 更新 memory.json

T+5min  标题同步完成（等待 run task + 读取 + 写入 Store）
T+5min  bridge.cleanup() 延迟清理队列
T+5min  run_mgr.cleanup() 延迟清理 RunRecord
```

---

> 本文档：端到端流程从 HTTP 请求经路由层 → Service 层 → 运行时层 → agent 构建 → ReAct 循环 → 推流收尾。五个核心对象各有不同生命周期。记忆更新延迟 30s 触发，标题同步和资源清理在后台延迟执行
