对话流程中的数据层——创建对话（Thread）、创建会话记录（RunRecord）、两个存储引擎（Checkpointer + Store）

---

## 一、两个存储引擎

系统使用两套独立的持久化引擎，各司其职：

| 引擎 | LangGraph 接口 | 存什么 | 读场景 | 写场景 |
|------|---------------|--------|--------|--------|
| **Checkpointer** | `BaseCheckpointSaver` | 完整的 agent 状态快照（messages、title、artifacts 等所有 channel_values） | 读取对话历史、恢复中断点、获取最新状态 | 每轮 ReAct 循环自动写检查点 |
| **Store** | `BaseStore` | 轻量元数据（thread_id、status、created_at、metadata、title） | `/threads/search` 快速列表、列表页展示 | 创建 thread 时写、run 结束后同步 title |

### 为什么两个？

- **Checkpointer 是重状态**：存整个 LangGraph 图的 channel_values（消息数组、产物列表等），数据量大，读写代价高
- **Store 是轻索引**：只存元数据字典，支持 namespace 搜索（`asearch`），适合列表页快速查询
- 列表页不需要加载每个 thread 的完整消息历史，只需要标题、时间、状态——Store 就是为此优化的快速索引

### 后端类型

两者共享 `config.yaml` 中 `checkpointer` 配置段，确保使用同一种持久化技术：

```yaml
checkpointer:
  type: sqlite            # memory | sqlite | postgres
  connection_string: store.db
```

| type | Checkpointer 实现 | Store 实现 |
|------|-------------------|-----------|
| `memory` | `InMemorySaver` | `InMemoryStore` |
| `sqlite` | `AsyncSqliteSaver` | `AsyncSqliteStore` |
| `postgres` | `AsyncPostgresSaver` | `AsyncPostgresStore` |

初始化入口在 `deps.py` 的 `langgraph_runtime()`：

```python
app.state.checkpointer = await stack.enter_async_context(make_checkpointer())
app.state.store         = await stack.enter_async_context(make_store())
```

文件位置：
- Checkpointer 工厂：`packages/harness/deerflow/agents/checkpointer/async_provider.py`
- Store 工厂：`packages/harness/deerflow/runtime/store/async_provider.py`

## 二、创建对话（Thread）

**路由**：`POST /api/threads` — `threads.py:create_thread()`

### 流程

```
前端 POST /api/threads { metadata: {...} }
  │
  ├─ 1. 生成 thread_id（uuid4）或使用前端传入的
  ├─ 2. 检查 Store 中是否已存在（幂等性）
  │     └─ 已存在 → 直接返回已有记录
  ├─ 3. 写入 Store（轻量元数据）
  │     └─ store.aput(namespace=("threads",), key=thread_id, value={...})
  └─ 4. 写入空 Checkpoint（让 state 接口立即可用）
        └─ checkpointer.aput(config, empty_checkpoint(), metadata, {})
```

### Store 中保存的 Thread 记录结构

```python
{
    "thread_id": "uuid-string",
    "status": "idle",              # idle | busy | interrupted | error
    "created_at": 1715760000.0,    # time.time() 时间戳
    "updated_at": 1715760000.0,
    "metadata": {},                 # 前端传入的自定义元数据
    "values": {}                    # 后续同步的 {"title": "..."}
}
```

### Checkpoint 中保存的初始状态

```python
# 空检查点，step=-1，无 channel_values
config = {"configurable": {"thread_id": "uuid", "checkpoint_ns": ""}}
empty_checkpoint() → { channel_values: {}, versions: {}, ... }
metadata = {"step": -1, "source": "input", "writes": None, "parents": {}}
```

### 幂等性

`create_thread` 是幂等的：如果 Store 中已有该 `thread_id`，直接返回已有记录，不重复写入。前端可以安全重试

## 三、创建会话记录（RunRecord）

RunRecord 不是持久化到数据库的，而是**内存中的运行时注册表**，由 `RunManager` 管理。

**文件**：`packages/harness/deerflow/runtime/runs/manager.py`

### RunRecord 数据结构

```python
@dataclass
class RunRecord:
    run_id: str                          # uuid4，每次运行唯一
    thread_id: str                       # 所属 thread
    assistant_id: str | None             # agent 标识
    status: RunStatus                    # pending → running → success/interrupted/error
    on_disconnect: DisconnectMode        # cancel | continue
    multitask_strategy: str              # reject | interrupt | rollback
    metadata: dict                       # 运行元数据
    kwargs: dict                         # {"input": ..., "config": ...}
    created_at: str                      # ISO 时间戳
    updated_at: str
    task: asyncio.Task | None            # 后台任务引用
    abort_event: asyncio.Event           # 取消信号
    abort_action: str                    # interrupt | rollback
    error: str | None
```

### RunStatus 状态机

```
pending → running → success
                  → interrupted（用户取消或被新 run 取代）
                  → error
```

### 创建入口

**路由**：`POST /api/threads/{thread_id}/runs/stream` → `services.py:start_run()`

```python
async def start_run(body, thread_id, request):
    # 1. 创建 RunRecord（含并发策略检查）
    record = await run_mgr.create_or_reject(
        thread_id, body.assistant_id,
        on_disconnect=disconnect,
        metadata=body.metadata,
        multitask_strategy=body.multitask_strategy,
    )

    # 2. 确保 thread 在 Store 中可见
    if store is not None:
        await _upsert_thread_in_store(store, thread_id, body.metadata)

    # 3. 准备参数，启动后台 task
    task = asyncio.create_task(run_agent(bridge, run_mgr, record, ...))
    record.task = task

    # 4. 异步同步标题到 Store
    asyncio.create_task(_sync_thread_title_after_run(task, thread_id, ...))

    return record
```

### 并发策略（multitask_strategy）

`create_or_reject()` 在 asyncio.Lock 保护下原子执行：

| 策略 | 同 thread 已有 running run 时 | 行为 |
|------|-------------------------------|------|
| `reject` | 抛出 `ConflictError` (409) | 拒绝新 run |
| `interrupt` | 设置 `abort_event` + `task.cancel()` | 中断旧 run，创建新 run |
| `rollback` | 同 interrupt，但运行时层会回滚到运行前检查点 | 中断并回滚 |

## 四、Thread 和 Run 的数据关系

```
Thread（Store + Checkpointer）
  │
  ├─ Store 记录：{thread_id, status, created_at, metadata, values.title}
  │   └─ 快速列表查询、列表页展示
  │
  ├─ Checkpointer 检查点链：ckpt_0 → ckpt_1 → ckpt_2 → ...
  │   └─ 每轮 ReAct 循环写入一个检查点
  │   └─ 完整的 channel_values（messages、artifacts 等）
  │   └─ 支持历史回溯、中断恢复
  │
  └─ RunRecord（内存，不持久化）
      └─ 每次对话一轮 run，run 结束后延迟 cleanup
      └─ 运行时状态追踪（pending/running/success/error）
      └─ 取消信号（abort_event）
```

### 关键区别

| | Thread | Run |
|---|--------|-----|
| 生命周期 | 持久化，手动删除才消失 | 运行期间存在，结束后延迟清理 |
| 存储位置 | Store（元数据）+ Checkpointer（状态） | RunManager 内存字典 |
| 一对多 | 一个 Thread 可以有多次 Run | 一个 Run 属于一个 Thread |
| 用途 | 对话会话的容器 | 一次对话执行的运行时记录 |

## 五、数据写入时序

一次完整的对话交互中，数据的写入顺序：

```
前端发起对话
  │
  ├─ [1] POST /api/threads                    → Store 写入 thread 元数据
  │                                          → Checkpointer 写入空检查点
  │
  ├─ [2] POST /api/threads/{id}/runs/stream   → RunManager 创建 RunRecord（内存）
  │                                          → Store upsert thread（确保可见）
  │
  ├─ [3] run_agent() 执行中                    → Checkpointer 每步自动写检查点
  │                                          → StreamBridge 推流（不持久化）
  │
  ├─ [4] run_agent() 结束                     → RunManager 设置最终状态
  │                                          → StreamBridge 发 end 哨兵
  │
  └─ [5] 标题同步任务                          → Checkpointer 读取 title
                                             → Store 更新 values.title
```

### 数据一致性

- **Store 和 Checkpointer 之间是最终一致**：run 结束后异步同步 title 到 Store
- **RunRecord 是临时状态**：不持久化，重启后丢失，但不影响——检查点已保存在 Checkpointer 中
- **Store 作为 Checkpointer 的快速索引**：search 时先查 Store（快），再补充 Checkpointer 中未索引的 thread（Phase 2 懒迁移）

## 六、Store 的两阶段搜索

`POST /api/threads/search` — `threads.py:search_threads()`

```
Phase 1: Store 快速查询
  └─ store.asearch(namespace=("threads",), limit=10000)
  └─ 返回所有已知 thread 元数据
  └─ O(threads) 但数据量小（只存元数据）

Phase 2: Checkpointer 补充（懒迁移）
  └─ 遍历 checkpointer.alist() 发现 Store 中缺失的 thread
  └─ 立即写入 Store（懒迁移，下次搜索不需要再查）
  └─ 确保 Store 最终收敛为完整索引

Phase 3: 过滤 → 排序 → 分页
  └─ 按 metadata 精确匹配过滤
  └─ 按 updated_at 倒序排列
  └─ offset + limit 分页
```

Phase 2 解决的边界情况：通过 LangGraph Server 直接创建的 thread（没走 Gateway 的 `POST /threads`），Gateway 的 Store 中没有记录。Phase 2 在搜索时发现并自动补录

## 七、ThreadState（Agent 运行时状态）

**文件**：`packages/harness/deerflow/agents/thread_state.py`

这是 LangGraph 图内部的状态模式，存在 Checkpointer 的 `channel_values` 中，**不是**独立数据库表：

```python
class ThreadState(AgentState):
    sandbox: SandboxState | None           # 沙箱 ID
    thread_data: ThreadDataState | None    # workspace/uploads/outputs 路径
    title: str | None                      # 自动生成的标题
    artifacts: list[str]                   # 产物路径（自动去重）
    todos: list | None                     # 待办列表
    uploaded_files: list[dict] | None      # 上传文件元数据
    viewed_images: dict[str, ViewedImageData]  # 图片缓存
```

其中 `artifacts` 和 `viewed_images` 使用 Annotated 的 reducer 函数实现自动归并去重，确保多轮执行中不会丢失已有数据

---

> 本文档：对话流程涉及两套存储引擎——Checkpointer 存完整 agent 状态快照，Store 存轻量元数据做快速索引。创建 Thread 时同时写入两者，创建 RunRecord 是纯内存操作。Run 结束后异步将 title 从 Checkpointer 同步到 Store。搜索时先查 Store（快），再用 Checkpointer 补充未索引的 thread（懒迁移）
