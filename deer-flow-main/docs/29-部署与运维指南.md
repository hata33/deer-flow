# 【29】部署与运维指南

## 1. 部署架构概览

DeerFlow采用**微服务架构**，包含以下核心服务：

| 服务 | 端口 | 职责 |
|------|------|------|
| **Frontend** | 3000 | Next.js Web界面 |
| **Gateway API** | 8001 | FastAPI RESTful接口 |
| **LangGraph Server** | 2024 | LangGraph代理运行时 |
| **Nginx** | 2026 | 统一反向代理入口 |
| **Provisioner** | 8002 | 沙箱资源管理（可选） |

## 2. 本地开发环境

### 2.1 环境要求

- **Node.js**：22+（前端）
- **Python**：3.12+（后端）
- **pnpm**：10.26.2+（前端包管理）
- **uv**：最新版（后端包管理）

### 2.2 快速启动

```bash
# 克隆项目
git clone https://github.com/bytedance/deer-flow.git
cd deer-flow

# 安装依赖
make install

# 启动所有服务
make dev
```

访问 `http://localhost:2026`

### 2.3 分服务启动

```bash
# 终端1：LangGraph Server
cd backend
make dev

# 终端2：Gateway API
cd backend
make gateway

# 终端3：Frontend
cd frontend
pnpm dev
```

## 3. 生产部署

### 3.1 Docker Compose部署

**配置文件**：`docker-compose.yml`

```yaml
services:
  langgraph:
    build:
      context: ./backend
      dockerfile: Dockerfile.langgraph
    ports:
      - "2024:2024"

  gateway:
    build:
      context: ./backend
      dockerfile: Dockerfile.gateway
    environment:
      - DEER_FLOW_LANGGRAPH_URL=http://langgraph:2024
    depends_on:
      - langgraph

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    environment:
      - NEXT_PUBLIC_LANGGRAPH_BASE_URL=/api/langgraph
    depends_on:
      - gateway

  nginx:
    image: nginx:alpine
    ports:
      - "2026:2026"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - frontend
```

**启动命令**：
```bash
docker-compose up -d
```

### 3.2 生产环境配置

#### 环境变量

```bash
# LangGraph Server
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=your-key

# Gateway API
DEER_FLOW_LANGGRAPH_URL=http://langgraph:2024
DEER_FLOW_CHANNELS_LANGGRAPH_URL=http://langgraph:2024

# Frontend
NEXT_PUBLIC_LANGGRAPH_BASE_URL=/api/langgraph
```

#### 配置文件

- **config.yaml**：主配置文件，定义模型、沙箱、工具等
- **extensions_config.json**：扩展配置，定义MCP服务器与技能状态

### 3.3 健康检查

```bash
# Gateway健康检查
curl http://localhost:8001/health

# LangGraph健康检查
curl http://localhost:2024/health

# Nginx健康检查
curl http://localhost:2026/health
```

## 4. 监控与日志

### 4.1 日志配置

**日志级别**：通过`config.yaml`的`log_level`配置
- `debug`：详细调试信息
- `info`：常规运行信息（默认）
- `warning`：警告信息
- `error`：错误信息

**日志输出**：控制台标准输出，Docker环境通过`docker logs`查看

### 4.2 性能监控

**LangSmith追踪**：
```yaml
# config.yaml
tracing:
  enabled: true
  project_name: "deer-flow-production"
```

### 4.3 Token使用追踪

```yaml
# config.yaml
token_usage:
  enabled: true
```

**日志输出**：
```
INFO:deerflow.agents.middlewares.token_usage: Model claude-sonnet-4.6: input=1234, output=5678, total=6912 tokens
```

## 5. 运维操作

### 5.1 配置更新

**热更新支持**：
- 模型配置：重启服务生效
- MCP配置：通过Gateway API实时更新
- 技能启用/禁用：通过Gateway API实时更新
- 记忆数据：通过Gateway API重新加载

### 5.2 数据备份

**重要数据**：
- `backend/.deer-flow/memory.json`：用户记忆数据
- `backend/.deer-flow/threads/`：线程本地数据
- `extensions_config.json`：MCP与技能配置

**备份策略**：
```bash
# 定期备份
tar -czf deer-flow-backup-$(date +%Y%m%d).tar.gz \
  backend/.deer-flow/ \
  extensions_config.json \
  config.yaml
```

### 5.3 故障排查

**常见问题**：

1. **服务启动失败**
   ```bash
   # 检查端口占用
   lsof -i :8001
   lsof -i :2024

   # 查看服务日志
   docker-compose logs gateway
   docker-compose logs langgraph
   ```

2. **模型调用失败**
   ```bash
   # 检查API密钥配置
   cat config.yaml | grep api_key

   # 测试模型连接
   curl -X POST http://localhost:2024/invoke \
     -H "Content-Type: application/json" \
     -d '{"model": "claude-sonnet-4.6", "messages": [{"role": "user", "content": "test"}]}'
   ```

3. **技能加载失败**
   ```bash
   # 检查技能目录
   ls -la skills/public/
   ls -la skills/custom/

   # 验证SKILL.md格式
   python -m deerflow.skills.validation skills/public/data-analysis/SKILL.md
   ```

### 5.4 日志级别调整

**临时调整**：
```bash
LOG_LEVEL=debug make dev
```

**永久调整**：
```yaml
# config.yaml
log_level: debug
```

## 6. 安全建议

### 6.1 API密钥管理

- **使用环境变量**：避免将密钥硬编码到配置文件
- **定期轮换**：定期更新API密钥
- **最小权限**：只为需要的用户/服务分配密钥

### 6.2 网络安全

- **内网部署**：LangGraph与Gateway服务部署在内网
- **反向代理**：通过Nginx统一入口，便于配置SSL与认证
- **防火墙规则**：只暴露必要端口（如2026）

### 6.3 文件系统安全

- **目录隔离**：每个线程使用独立目录，防止路径遍历
- **权限控制**：限制`.deer-flow/`目录访问权限
- **沙箱隔离**：使用Docker沙箱隔离文件系统访问

## 7. 扩展部署

### 7.1 负载均衡

**多实例部署**：
- 启动多个Gateway实例
- 使用Nginx负载均衡

```nginx

    server gateway1:8001 max_fails=3 fail_timeout=30s;
    server gateway2:8001 max_fails=3 fail_timeout=30s;
}
```

**数据持久化**：
- 使用PostgreSQL替代JSON文件存储
- 使用Redis缓存会话状态
